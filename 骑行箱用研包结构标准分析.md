# 骑行箱用研包结构标准分析

## 文档 1：骑行箱用研座谈会

```yaml
文档名: 骑行箱用研座谈会
文档类型: 逐字稿（座谈会录音转写）
总字数: 约67,000字（67,121字符）
结构分析:
  - 一级标题: 无明确一级标题，为连续对话流
  - 二级标题: 无
  - 关键模块:
    - 关键词标签区（顶部自动提取）
    - 发言人编号+时间戳标注
    - 完整对话记录（2小时44分钟）
    - 涵盖：自我介绍、购买动机、使用场景、路面适应性、满意度评分、痛点排序、新品概念体验、价格预期、品牌偏好
格式特征:
  - 表格: 否
  - 评分: 是（用户当场打分，如8-9分满意度）
  - 用户原话: 是（核心价值，完整保留每位用户的真实表达）
  - 数据: 是（价格、续航里程、重量等具体数字）
  - 时间线: 是（按时间戳推进，从开场到结束）
内容质量:
  - 信息密度: 高（2.7小时完整对话，信息量极大）
  - 结构化程度: 低（纯原始转写，未做任何结构化整理）
  - 可操作性: 低（需二次加工才能用于决策）
核心模式:
  - 独特价值: 用户真实原话的完整保留，是所有后续分析的一手素材来源
  - 与其他文档的关系: 是会议纪要（文档2）的原始输入；访谈结果（文档5）的数据来源之一
```

## 骑行箱用研包结构标准

基于以上 7 份文档的深度分析，提炼出用研包的完整结构标准：

### 一、用研包的标准组成（7类文档）

| 序号 | 文档类型 | 核心作用 | 结构化程度 | 信息密度 |
|------|---------|---------|-----------|---------|
| 1 | 逐字稿 | 一手素材，用户原话完整保留 | 低 | 高 |
| 2 | 会议纪要 | 快速回顾，共识+待办提炼 | 高 | 高 |
| 3 | 用研需求文档 | 方法论骨架，定义"为什么"和"怎么问" | 高 | 高 |
| 4 | 招标文档 | 项目管理，连接需求与执行 | 高 | 中高 |
| 5 | 最终报告 | 分析产出，可直接指导产品定义 | 高 | 高 |
| 6 | 问卷数据 | 原始数据，支撑定量结论 | 低（需嵌入文件） | 视数据量 |
| 7 | 访谈展示材料 | 视觉引导，激发用户反馈 | 中 | 中 |

### 二、每类文档的结构规范

#### 2.1 逐字稿
- **必备元素**：时间戳、发言人标识、关键词提取区
- **格式要求**：纯文本对话流，按时间顺序排列
- **核心价值**：不可替代的原始素材，所有分析的根基

#### 2.2 会议纪要
- **必备元素**：会议元信息（时间/参会人/录屏）、共识提炼、核心议题与策略对照表、待办清单（含责任人）、章节索引
- **格式要求**：四模块结构（总结→议题→待办→章节）
- **核心价值**：将数小时对话浓缩为可执行的决策依据

#### 2.3 用研需求文档
- **必备元素**：调研背景与目的、商业问题矩阵、调研形式与人群定义、问卷/访谈工具设计
- **格式要求**：从目标→问题→方法→工具的完整链条
- **核心价值**：定义整个用研项目的"骨架"

#### 2.4 招标文档
- **必备元素**：背景与目的、商业问题（比需求文档更聚焦验证）、样本量与方法要求、时间节点、交付形式、招标要求、报价模板
- **格式要求**：标准招标文档结构
- **核心价值**：将内部需求转化为可执行的外部协作规格

#### 2.5 最终报告
- **必备元素**：用户基本情况、用户画像矩阵、购买决策分析、体验反馈（满意/不满意）、升级期待、创新方向与建议
- **格式要求**：大量结构化表格+提及数统计+典型用户原话
- **核心价值**：将定性洞察转化为可量化的结论，直接指导产品定义

#### 2.6 问卷数据
- **必备元素**：原始数据文件（Excel/CSV）、数据说明
- **格式要求**：嵌入式或独立文件
- **核心价值**：支撑定量结论的原始证据

#### 2.7 访谈展示材料
- **必备元素**：市面产品介绍（参数+图片）、竞品外观对比、概念产品定义
- **格式要求**：图文结合，以图片为主
- **核心价值**：访谈中的视觉引导工具

### 三、文档间的逻辑关系

```
用研需求文档（骨架）
  ↓ 指导
招标文档 → 外部供应商执行
  ↓ 产出
问卷数据（定量） + 逐字稿（定性）
  ↓ 加工
会议纪要（快速回顾） + 最终报告（深度分析）
  ↓ 输入
产品定义 / OnePage / 功能说明书
```

### 四、质量检查清单

- [ ] 逐字稿是否完整保留用户原话，含时间戳和发言人标识
- [ ] 会议纪要是否包含共识、待办（含责任人）和章节索引
- [ ] 需求文档是否覆盖完整的商业问题矩阵和工具设计
- [ ] 招标文档是否包含样本量、时间节点和招标要求
- [ ] 最终报告是否用提及数量化结论，保留典型用户原话
- [ ] 展示材料是否包含竞品参数对比和概念产品定义
- [ ] 所有文档是否可通过来源链路追溯到原始素材
