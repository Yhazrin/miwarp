# ADR-007: v1.0.9 Contract Test Architecture

- **Status**: Accepted
- **Date**: 2026-06-22
- **Context**: Branch `test/v109-release-quality`
- **Supersedes**: n/a

## Context

v1.0.9 introduces three frozen contracts (Runtime Hub, Transaction
Contracts, Shared File Ownership) plus a release-quality gate that
the integrator runs before tagging. The contracts are
forward-looking — the implementation lands in Wave 1+ (Agent A/B/C/D
branches) — but the contract TESTS need to exist on the
integration branch today so:

1. The spec doc is pinned (no silent drift).
2. The future implementation has a clear pass/fail target.
3. The release-quality gate has a single command (`arch:check`)
   that the CI runs on every PR.

The existing architecture-gate pattern (ADR-002, ADR-003) uses
text-only parsers in `scripts/architecture/`. v1.0.9 extends that
pattern with three new scripts and a per-commit file-budget
guardrail.

## Decision

Add the following test infrastructure:

1. **`scripts/architecture/cross-platform-bus-contract.mjs`** —
   cross-checks `runtime_hub_*` and `diagnostics_*` Tauri handlers
   against the iOS WS dispatch allowlist, and the
   `BusEvent::Runtime*` / `BusEvent::Diagnostic*` variants against
   the frontend `BusEvent` union. Dormant when no implementation
   exists; activates as Agent B/D land.

2. **`scripts/architecture/file-budget-diff.mjs`** — per-commit
   file-size budget check. Fails if any added/modified file in the
   diff range exceeds the fail threshold. Distinct from
   `file-budget.mjs` (which checks the final state); this is the
   pre-commit guard.

3. **`scripts/architecture/__tests__/runtime-contract.test.ts`** —
   Vitest test that parses the spec doc and asserts the structural
   shape (4 commands, 12 capabilities, full §3/§9/§4 type
   coverage). Dormant assertions activate when the implementation
   lands.

4. **Rust fault-injection harnesses** at `src-tauri/tests/`:
   - `send_fault_injection.rs` — 15 tests pinning the
     SendTransaction state machine and accepted_client_message_ids
     ledger.
   - `permission_fault_injection.rs` — 13 tests pinning the
     PermissionTransaction wire format and NEVER_ALLOW_TOOLS.
   - `runtime_contract_types.rs` — 6 tests pinning the BusEvent
     wire format for the v1.0.9 runtime/diagnostic namespace.

5. **E2E golden path** at `e2e/golden-path.spec.ts` — Playwright
   spec driving the desktop app through 12 release-quality steps
   (open → send → respond → permission → tool → switch →
   diagnostics → export zip). Optional peer dep; dormant on
   runners without Playwright.

6. **`.github/workflows/contract-tests.yml`** — separate CI lane
   that runs the v1.0.9-specific gates alongside the main
   `ci.yml`. Triggered by paths filter on contract surface files.

7. **Documentation**:
   - `docs/architecture/v1.0.9-release-checklist.md` — 10-step
     pre-tag verification.
   - `docs/architecture/v1.0.9-rollback-checklist.md` — rollback
     decision criteria and procedure.
   - `docs/architecture/cross-platform-capability-matrix.md` —
     extended with §7 "v1.0.9 Migration Compatibility Matrix".

8. **Frontend dormant tests**:
   - `src/lib/runtime/__tests__/runtime-capabilities-contract.test.ts`
   - `e2e/__tests__/redaction-manifest-contract.test.ts`

9. **`package.json` script wiring**:
   - `arch:cross-platform-bus` → new script
   - `arch:runtime-contract` → new vitest invocation
   - `arch:budget:diff` → new script
   - `arch:check` extended to include the three new scripts
   - `arch:check:strict` → `arch:check` + `arch:budget:diff`
   - `test:contract` → convenience for the two contract test files

## Consequences

- **Forward-looking enforcement**: the contract tests are dormant
  on the integration HEAD (no `runtime_hub_*` code yet) and
  activate as Agent A/B/C/D land. A future change that adds a 5th
  `runtime_hub_*` command without an ADR fails the test.

- **Spec is the source of truth today**: the contract test
  parses the spec doc and asserts its internal consistency.
  Drift between the doc and the implementation is impossible
  to detect from the test (it tests the doc, not the code),
  but a manual code-review pass is the gate.

- **No source-code behavior change**: all new tests are additive.
  The existing `arch:check` baseline is preserved; the v1.0.9
  additions are appended to the chain.

- **CI is opt-in for the new lane**: the
  `contract-tests.yml` workflow triggers on paths filter, so
  PRs that don't touch the contract surface don't pay the
  cost. The main `ci.yml` is unchanged.

- **Test runtime is bounded**: the fault-injection harnesses
  run in < 1s each; the contract test is < 200ms; the
  per-commit budget check is < 1s. The whole v1.0.9 contract
  lane adds < 30s to CI.

## Alternatives Considered

- **A single mega-test file** that runs all the contract
  assertions — rejected. The current shape (one file per
  concern) is the same pattern as the existing
  `arch:check` scripts and makes the failure message
  unambiguous.

- **Generate contract tests from a schema** (e.g. JSON Schema
  → Vitest) — rejected. The spec doc is a Markdown file with
  hand-curated examples; generating tests from it would
  duplicate the parsing logic. The current approach
  (parse-and-assert) is what the codebase already does for
  the existing arch gate.

- **Add the contract assertions to `tauri-contract.mjs` and
  `ios-ws-contract.mjs`** — rejected. The task contract
  requires "extend existing scripts by adding NEW scripts,
  not changing existing ones." Adding to existing scripts
  would change their output and risk breaking dependent
  tooling.

- **A `tauri-driver` lane for E2E** instead of Playwright —
  considered for v1.0.10. Playwright is a faster onboarding
  path for the contract gate; `tauri-driver` would give
  real native-driver parity but requires more setup.
