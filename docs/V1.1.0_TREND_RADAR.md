# MiWarp v1.1.0 趋势雷达 · GitHub / 协议 / Agent 工程生态

> **文档状态**：Living Research Ledger
> **首次建立**：2026-06-23
> **目的**：持续记录外部技术信号，并判断是否进入 [`PLAN_V1.1.0.md`](./PLAN_V1.1.0.md)。
> **原则**：热度只用于发现，不用于决策；进入 MiWarp 必须同时满足用户价值、架构适配、可验证性和维护成本。

## 一、研究方法

### 1.1 信号等级

| 等级 | 含义                                             |
| ---- | ------------------------------------------------ |
| `S`  | 已形成标准、生态或强产品范式，应立即做兼容性设计 |
| `A`  | 高热且与 MiWarp 主线强相关，应进入锚点           |
| `B`  | 有价值，但应先实验或等待生态稳定                 |
| `C`  | 仅作观察，暂不投入                               |

### 1.2 采用判断

每个外部项目或技术按以下维度判断：

1. 是否解决 MiWarp 用户当前真实问题；
2. 是否能复用现有 Runtime Hub、BusEvent、Visual Block、Session、Diagnostics；
3. 是否能形成竞品难以快速复制的闭环；
4. 是否改善可靠性、性能、验证或安全；
5. 是否引入不可控云依赖；
6. 是否会迫使 MiWarp 重写核心；
7. 是否有活跃规范、实现、测试或跨客户端采用；
8. 能否在 v1.1.0 内形成明确 MVP。

### 1.3 决策类型

| 决策        | 含义                                    |
| ----------- | --------------------------------------- |
| `Adopt`     | 进入主计划并形成 Anchored / Frozen 锚点 |
| `Adapt`     | 吸收设计原则，但不直接依赖实现          |
| `Prototype` | 做受限原型，不进入主路径                |
| `Watch`     | 持续观察                                |
| `Reject`    | 当前明确不采用                          |

## 三、重点信号详记

## `R-001` Agent Client Protocol（ACP）

- **来源**：https://github.com/agentclientprotocol/agent-client-protocol
- **观察时间**：2026-06-23
- **外部状态**：协议稳定版本为 1；提供 Rust、TypeScript、Python、Java、Kotlin 等实现或 SDK；通过 initialize 中的 protocolVersion 和 capabilities 判断兼容性。
- **解决问题**：编码 Agent 与客户端之间长期存在 N×M 私有适配；ACP 试图统一会话、Prompt、工具、权限和能力协商边界。

### MiWarp 吸收点

- 不替换现有 Runtime Hub；
- 新建 ACP Adapter，将 ACP 消息映射到 Canonical Agent Events；
- 借鉴 capabilities negotiation，替代“配置存在即支持”的静态判断；
- 建立协议 fixture 和 conformance 测试；
- 为未来第三方 Agent 接入提供公开边界。

### 不采用点

- 不把内部 BusEvent 直接改造成 ACP wire schema；
- 不让协议版本决定内部领域模型；
- 不为 v1.1.0 承诺所有 ACP Agent 兼容。

**决策**：`Adapt + Prototype`

---

## `R-002` AG-UI

- **来源**：https://github.com/ag-ui-protocol/ag-ui
- **观察时间**：2026-06-23
- **核心信号**：轻量事件协议，定义约 16 类标准事件，支持实时聊天、双向状态、生成式 UI、上下文增强，并可运行在 SSE、WebSocket、Webhook 等传输上。

### MiWarp 吸收点

- 对照现有 BusEvent，建立 Canonical Event 分类和版本；
- 将 Conversation phase、tool、state、artifact 等映射到统一事件；
- 可选导出 AG-UI stream，方便未来 Web / Mobile / 第三方前端；
- 借鉴双向 state sync，但继续保留本地任务 Journal 为真实来源。

### 风险

- 过早直接依赖协议会使现有桌面链路受外部演进影响；
- AG-UI 面向通用 Agent UI，不包含 MiWarp 需要的 Git provenance、Worktree 和 Durable Run 语义。

**决策**：`Adapt + Prototype`

---

## `R-003` MCP Apps

- **来源**：https://github.com/modelcontextprotocol/ext-apps
- **观察时间**：2026-06-23
- **核心机制**：Tool 声明 `ui://` HTML 资源；Host 在 sandboxed iframe 中渲染；Host 把 Tool data 传给 UI，UI 可通过 Host 发起受控 Tool 调用。
- **生态信号**：官方仓库包含规范、SDK、App Bridge、Server Helper、多框架模板和大量交互示例。

### MiWarp 吸收点

- Visual Block 之外新增 MCP App Canvas；
- 统一 Conversation 中的交互式 Tool Result；
- 建立 iframe sandbox、CSP、权限、尺寸、主题、crash fallback；
- MCP App 的交互和后续 Tool call 进入权限层、Journal 和 Trace；
- 关键状态可快照成 Artifact。

### 边界

- Mermaid / Vega 等可信本地组件继续走 Visual Block；
- 第三方 MCP App 必须走更严格的隔离路径；
- 不允许 App 直接继承 CLI 或宿主文件权限。

**决策**：`Adopt`

---

## `R-004` MCP Registry

- **来源**：https://github.com/modelcontextprotocol/registry
- **观察时间**：2026-06-23
- **核心信号**：Registry 被定位为 MCP Server 的 app-store-like 发现服务；支持 GitHub OAuth / OIDC、DNS 和 HTTP 所有权验证；API 已经历 preview 和稳定窗口。

### MiWarp 吸收点

- Registry 作为发现源；
- 发布者和命名空间验证信息进入 Trust Center；
- 安装前做 schema、权限、命令、目录、网络和环境变量分析；
- 沙箱启动、tools/list、最小调用和 health probe；
- 更新前后 capability / permission diff。

### 关键判断

“已出现在 Registry”只表示可发现，不表示安全、质量或适配已经被 MiWarp 认证。

**决策**：`Adopt`

## `R-005` Playwright MCP 与 CLI + Skills 分流

- **来源**：https://github.com/microsoft/playwright-mcp
- **观察时间**：2026-06-23
- **核心信号**：通过 Accessibility Snapshot 而非纯截图操作网页；官方同时明确指出，高吞吐编码任务使用 CLI + Skills 更节省上下文，MCP 更适合持久状态、丰富 introspection 和长流程。

### MiWarp 吸收点

- Browser Verification Studio 采用 DOM / Accessibility Tree 为主、截图为证据；
- 稳定验证固化为 Playwright CLI / Script；
- 探索、修复和持久浏览器可使用 MCP 或 Agent loop；
- Context Budget Manager 统计浏览器 schema 和 snapshot 成本；
- 不把所有浏览器能力都塞成长期加载的 MCP tools。

**决策**：`Adapt`

## `R-006` Stagehand

- **来源**：https://github.com/browserbase/stagehand
- **观察时间**：2026-06-23
- **核心信号**：在代码和自然语言之间分工；预览 AI 动作；缓存可重复动作；稳定时无需 LLM，页面变化导致失败时再调用 AI 自愈。项目明确将可靠性、可扩展性、速度、成本作为优先顺序。

### MiWarp 吸收点

```text
第一次探索：Agent / natural language
        ↓
用户或系统确认动作
        ↓
固化为确定性 Playwright Step
        ↓
后续零模型重放
        ↓
页面变化失败时再请求 Agent 修复
```

- 将 browser steps 变成 Artifact；
- 将重放结果接入 Quality Gate；
- 预览高风险浏览器动作；
- 缓存命中和模型调用进入 Trace。

**决策**：`Adapt`

## `R-007` OpenCode

- **来源**：https://github.com/anomalyco/opencode
- **观察时间**：2026-06-23
- **核心信号**：开源编码 Agent 和桌面客户端竞争快速升温；内置 build、只读 plan 和 general subagent；支持大量模型与跨平台安装。

### MiWarp 吸收点

- Read Only / Workspace Write / Full Local 等 Execution Profile；
- Plan Gate 与 Build Run 明确分离；
- Runtime Control Tower 必须具备真实能力矩阵，而非只支持 Claude / Codex；
- Skill Packs 和 Recipe 比固定“Agent 人格”更适合工作流；
- MiWarp 的竞争点不能只是“又一个桌面聊天壳”，必须依靠 Task、Review、Verification、Artifact 和 Recovery 闭环。

### 不复制点

- 不复制完整终端 UI；
- 不因为热度立即接入全部 Provider；
- 不把 subagent 等同于完整多 Agent 产品。

**决策**：`Adapt`

## `R-008` Restate / Durable Execution

- **来源**：https://github.com/restatedev/restate
- **观察时间**：2026-06-23
- **核心信号**：可靠执行、部分进度恢复、可靠消息、持久 Promise / Timer、一致状态和可观测性被视为统一执行语义。

### MiWarp 吸收点

- Durable Run Journal；
- recovery cursor；
- action_id 与幂等分类；
- safe retry / manual confirmation / uncertain；
- Journal 与状态同步写入；
- Trace、Task 和恢复界面联动。

### 不采用点

- v1.1.0 不引入分布式工作流平台作为桌面运行硬依赖；
- 采用语义和不变量，不直接复制服务端架构。

**决策**：`Adapt`

## `R-009` Langfuse / OpenTelemetry

- **来源**：https://github.com/langfuse/langfuse
- **观察时间**：2026-06-23
- **核心信号**：Agent / LLM 工程正把 trace、指标、评估、数据集和 Prompt 管理统一；OpenTelemetry 提供跨服务语义和导出边界。

### MiWarp 吸收点

- Local Agent Trace；
- trace/span parent-child 模型；
- runtime、model、tool、permission、file、browser、recovery 属性；
- 本地可视化；
- 可选 OpenTelemetry 导出；
- 诊断包默认脱敏。

### 边界

- 默认不上传；
- 不把 Langfuse 服务作为 MiWarp 必需依赖；
- 不记录完整 Prompt、文件正文和 secret。

**决策**：`Adapt`

## `R-010` E2B / Daytona 沙箱

- **来源**：
  - https://github.com/e2b-dev/E2B
  - https://github.com/daytonaio/daytona
- **观察时间**：2026-06-23
- **核心信号**：隔离执行、端口暴露、快照、远程环境和 Agent 沙箱正成为独立基础设施层。

### MiWarp 吸收点

- `ExecutionEnvironment` 统一接口；
- Local、Worktree、Container、Remote Sandbox 均由 Adapter 提供；
- Task 显式选择执行环境；
- 端口、进程、文件、快照和销毁生命周期统一；
- Resource Governor 管理并发和回收。

### 决策

- Local Worktree 是 P0；
- Container 是 P1；
- E2B / Daytona 只做实验性 Adapter；
- 不把云沙箱变成默认路径。

**决策**：`Prototype`

## `R-015` MCP 供应链与权限风险

- **参考事件**：第三方 MCP Server 曾出现恶意版本窃取用户邮件的公开安全报道。
- **观察时间**：2026-06-23
- **核心信号**：MCP Server 通常接触高权限 API、数据库、文件和凭据，但传统资产清单、供应链审计和权限可视化尚不成熟。

### MiWarp 吸收点

- 来源与发布者验证；
- package version pin；
- hash / lockfile；
- install / update permission diff；
- 运行网络和工具行为 Trace；
- 熔断和 quarantine；
- Credential scope；
- 明确风险标签。

**决策**：`Adopt`

## `R-016` Serena / LSP Semantic Tools

- **来源**：https://github.com/oraios/serena
- **观察时间**：2026-06-23
- **核心信号**：以 Language Server 为基础，为 Agent 提供 symbol outline、definition、references、rename、symbol body replacement 等高层语义工具；强调大型代码库中的速度、token 效率和结构化编辑。

### MiWarp 吸收点

- 将 Semantic Code Intelligence 作为本地能力层，而不是只作为某个 MCP Server 的特性；
- 复用 Language Server，建立可插拔语言后端；
- Context Manager 优先注入 symbol、引用关系和相关代码片段；
- Agent 执行 rename / refactor 前展示影响范围；
- 结构化编辑继续进入 Diff provenance、Journal 和 Trace；
- 普通文本工具保留为降级路径。

### 风险

- 不同语言服务器能力不一致；
- 长驻 LSP 会增加内存和进程治理成本；
- Workspace 文件变化、分支切换和 Worktree 需要独立索引边界。

**决策**：`Adapt + Prototype`

- **来源**：https://github.com/ast-grep/ast-grep
- **观察时间**：2026-06-23
- **核心信号**：AST 级结构搜索、lint 和 rewrite 可以用接近代码的模式表达；Rust + Tree-sitter + 多核适合本地高性能静态分析和 codemod。

### MiWarp 吸收点

- 为 Agent 提供 structural search 而非只靠 regex；
- 批量修改先 dry-run，显示匹配数量和影响文件；
- 将 AST rule 接入 Quality Gate；
- 为依赖升级、API 迁移和重复异味治理提供 Recipe；
- 结果作为 Artifact，可复查规则、匹配和改写。

### 边界

- 不把所有编辑强制转为 AST；
- 语言支持和 parser 失败时明确降级；
- 自动 rewrite 仍需权限、Diff 和验证。

**决策**：`Adapt`

## `R-018` GitHub Spec Kit

- **来源**：https://github.com/github/spec-kit
- **观察时间**：2026-06-23
- **核心信号**：将项目原则、需求说明、技术计划、任务拆分、跨制品一致性分析、实现和收敛检查组织成标准化 Spec-Driven Development 流程，并已适配大量编码 Agent。

### MiWarp 吸收点

- Task 前增加可选 Spec / Clarification / Plan；
- 将 acceptance criteria 直接绑定 Quality Gate；
- Spec、Plan、Tasks、Checklist 作为 Artifact；
- 提供 converge 检查，比较需求、实现和测试之间的缺口；
- 用 Skill Pack / Recipe 分发流程，不把某个外部 CLI 变成硬依赖。

### 关键取舍

- 轻量 bug fix 不强制走完整 SDD；
- MiWarp 只吸收结构化流程和可追踪关系；
- 不把文档生成数量当成质量。

**决策**：`Adapt`

## `R-019` Repomix / Agent Context Packaging

- **来源**：https://github.com/yamadashy/repomix
- **观察时间**：2026-06-23
- **核心信号**：仓库结构、技术栈、文件内容、token 预算和压缩可以打包为 Agent-friendly 输出，并可生成可安装 Skill。

### MiWarp 吸收点

- Project Context Pack 显示目录、技术栈、关键文件和 token 预算；
- 支持按范围、忽略规则和 Task 生成 pack；
- pack 超过预算时失败或拆分，而不是静默截断；
- 优先生成 Manifest + Artifact pointers，不默认把全仓复制进模型；
- 可导出为跨 Agent Skills 兼容包。

### 边界

- 全仓打包只适合显式导出或小仓库；
- 日常运行优先 Semantic Code Intelligence 和按需检索；
- 必须遵循 secret、binary、ignore 和 PII 规则。

**决策**：`Adapt`

## `R-020` GitButler

- **来源**：https://github.com/gitbutlerapp/gitbutler
- **观察时间**：2026-06-23
- **核心信号**：围绕 AI 工作流提供 stacked branches、parallel branches、拖拽式 commit mutation、Undo Timeline、first-class conflicts 和 forge integration；桌面端同样采用 Tauri、Svelte、TypeScript 与 Rust。

### MiWarp 吸收点

- 把并行 Agent 的修改显示为独立 change lane；
- Task 不只绑定 Worktree，也绑定可解释的 change stack；
- 所有 Git / 文件操作进入 Undo Timeline；
- 支持 split、move、squash、reword 等 Review 收口操作；
- 冲突作为可保存、可稍后处理的状态，而不是任务立即死亡；
- 研究其 Tauri / Svelte / Rust 大型桌面应用的性能与模块边界。

### 许可与边界

- GitButler 当前采用 Fair Source 许可并限制竞争性复制；
- MiWarp 仅吸收公开的高层交互范式和产品原则，不复制代码、内部实现或受限资产；
- 默认仍使用标准 Git / Worktree。

**决策**：`Adapt`

## `R-021` Jujutsu VCS

- **来源**：https://github.com/jj-vcs/jj
- **观察时间**：2026-06-23
- **核心信号**：working copy 由真实 commit 表示；命令执行前自动 snapshot；所有 repo operation 与操作后状态被记录，可逐步 undo 或恢复任意 operation；冲突可以记录在 commit 中并稍后解决。

### MiWarp 吸收点

- Durable Run Journal 与 Git operation 形成统一 operation identity；
- 每次 Agent 改动前后建立可恢复 snapshot；
- Undo 不只针对最后一步，可基于依赖图回退特定 operation；
- conflicted change 作为 Task 状态保存，不强迫立即中断整个 Run；
- 研究可选 `jj` Adapter，但不在 v1.1.0 替换 Git。

### 风险

- Jujutsu 尚未达到 1.0，on-disk format 与工作流仍可能变化；
- Git / jj colocated workspace 需要严格能力探测和降级；
- 不能让用户在不理解语义的情况下自动转换仓库。

**决策**：`Adapt + Prototype`

## `R-022` Development Containers

- **来源**：https://github.com/devcontainers/spec
- **观察时间**：2026-06-23
- **核心信号**：用开放规范描述完整开发环境，可复用 Docker Compose 或单容器，并可将同一配置用于本地开发、CI 构建和测试。

### MiWarp 吸收点

- 原生识别 `devcontainer.json` 和 Features；
- Environment Capsule 可引用现有规范，不再发明第二套容器描述；
- Task 可在同一环境中本地执行、验证和生成 CI 建议；
- 环境缺失、端口、mount、feature 和 lifecycle command 进入可观察清单。

### 边界

- 不要求每个项目创建 Dev Container；
- 不自动执行未经审查的 lifecycle commands；
- secret 只保存引用，不写入 Artifact。

**决策**：`Adopt`

## `R-023` Dagger

- **来源**：https://github.com/dagger/dagger
- **观察时间**：2026-06-23
- **核心信号**：将交付工作流做成 programmable、local-first、repeatable、observable 的类型化执行图；支持 content-addressed typed artifacts、增量执行、自动缓存和 OpenTelemetry trace，可在本地、CI、沙箱或云环境运行。

### MiWarp 吸收点

- Environment Capsule 不只是镜像，而是可观察的 execution graph；
- 验证步骤按输入建立 cache key，只重新运行受影响节点；
- Build、Test Report、Package 等成为 typed Artifact；
- 运行图与 Local Agent Trace 对齐；
- Dagger 可作为可选 Recipe backend 原型，不成为桌面端硬依赖。

### 风险

- 容器运行时、镜像缓存和跨平台磁盘成本较高；
- 不应为了通用 DAG 把简单本地命令复杂化；
- 必须保留直接 CLI 路径。

**决策**：`Adapt + Prototype`

## `R-024` Letta Memory Blocks

- **来源**：https://github.com/letta-ai/letta
- **观察时间**：2026-06-23
- **核心信号**：stateful agent 通过显式 `memory_blocks` 保存可长期使用的结构化内容，并支持本地 CLI、Skills 和 subagents。

### MiWarp 吸收点

- 将项目长期知识拆成少量命名 blocks，而不是隐藏的向量记忆；
- 用户可以看到、编辑、锁定和拒绝记忆；
- block 有来源、更新时间和适用范围；
- Agent 修改 memory 必须经过 diff / approval；
- 相关 block 按 Task 和 token budget 注入。

### 不采用点

- 不允许 Agent 静默自我改写长期人格或项目事实；
- 不把完整聊天历史永久注入；
- Personal AI 仍不进入 v1.1.0 主线。

**决策**：`Adapt`

## `R-025` Promptfoo

- **来源**：https://github.com/promptfoo/promptfoo
- **观察时间**：2026-06-23
- **核心信号**：提供本地 Prompt / Agent / RAG 评测、模型并排比较、CI/CD 检查和 red teaming；强调数据驱动而非凭感觉判断模型与 Prompt 质量。

### MiWarp 吸收点

- 将真实 Task 保存为 repo-specific Eval Case；
- 比较 Runtime、Model、Prompt、Skill、Context Strategy；
- 评测结果可进入 release gate；
- 对 MCP、权限和工具描述做越权、注入和 secret 泄露测试；
- 结果默认本地保存，必要时导出脱敏报告。

### 边界

- 不将 Promptfoo 设为内置硬依赖；
- 可优先实现兼容导入 / 导出和受限 Adapter；
- 自动评分必须能回到具体执行证据。

**决策**：`Adapt`

## `R-026` MCP Capability Attestation Research

- **来源**：
  - https://arxiv.org/abs/2601.17549
  - https://arxiv.org/abs/2602.01129
- **观察时间**：2026-06-23
- **核心信号**：近期研究集中指出 MCP 的 capability 自述、双向 sampling、跨 Server 信任传播、tool poisoning 和供应链完整性仍存在协议级风险；建议加入来源认证、能力证明、细粒度策略和审计链。

### MiWarp 吸收点

- Registry metadata 不等于 capability attestation；
- 对 Server package、manifest、tool schema 和 App resource 保存哈希与版本；
- tool description 改变后旧授权失效；
- sampling、network、filesystem、secret 和 UI capability 分别授权；
- origin 与 user intent 贯穿每个 Tool call；
- Policy Engine 负责强制，不把安全寄托在 Prompt。

**决策**：`Adapt`

## 五、明确不跟随的热点

| 热点                            | 当前决策          | 原因                                                 |
| ------------------------------- | ----------------- | ---------------------------------------------------- |
| 无限制多 Agent swarm            | Reject for v1.1.0 | 难以验证、资源失控、当前隔离不足                     |
| 全视觉浏览器点击                | Reject as default | 不稳定、昂贵、难重放；结构化 DOM 优先                |
| 所有工具全部 MCP 化             | Reject            | schema/context 成本高，CLI/Skills 对高吞吐任务更合适 |
| 默认云沙箱                      | Reject            | 违背 local-first，增加隐私和成本依赖                 |
| 自动 Push / 自动 Merge 默认开启 | Reject            | 高风险且破坏用户控制                                 |
| Registry 一键无审计安装         | Reject            | 发现不等于信任                                       |
| 用协议替换内部全部模型          | Reject            | 外部标准仍在演进，必须通过 Adapter 隔离              |
| 只按 GitHub Star 决定路线       | Reject            | Star 不能证明产品适配和可靠性                        |
