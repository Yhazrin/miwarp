# MiWarp Mobile — Pairing & Security Model

## Overview

MiWarp Mobile connects to a MiWarp Desktop instance over a local network via WebSocket. The security model is designed to be safe by default while remaining practical for same-network use.

## Threat Model

### In Scope

- Unauthorized access from devices on the same LAN
- Token exposure through logs, screenshots, or insecure storage
- Accidental public network exposure
- Permission escalation through auto-approval

### Out of Scope (for v1)

- End-to-end encryption (relies on network-level security)
- Cloud relay / NAT traversal (use SSH tunnel or VPN)
- Multi-user access control (single token per desktop)

## Token Security

### Generation

- Token is 32 characters, alphanumeric (a-z, A-Z, 0-9)
- Generated using `rand::thread_rng()` (CSPRNG)
- Generated at desktop startup and stored in memory
- Can be regenerated by the user via Settings > Web Server > Mobile Access

### Storage on Mobile

| Platform | Storage Method | Notes |
|----------|---------------|-------|
| iOS | Keychain (kSecClassGenericPassword) | Hardware-backed, not included in iCloud backup |
| Android | EncryptedSharedPreferences | AES-256-GCM, keys in AndroidKeyStore (hardware-backed) |

### What is NEVER stored

- Token is never written to application logs
- Token is never stored in UserDefaults (iOS) or SharedPreferences (Android)
- Token is never included in crash reports
- Token is never sent to any analytics or third-party service

### Token in Transit

- Token is passed as a URL query parameter: `ws://{host}:{port}/ws?token={token}`
- This is visible to network-level observers on the same LAN
- For untrusted networks, users MUST use a tunnel (ngrok, cloudflared) or VPN
- The desktop does not support TLS natively — the query string is plaintext

## Pairing Flow

### Manual Pairing

1. User enters host, port, and token manually on mobile.
2. Mobile connects to `ws://{host}:{port}/ws?token={token}`.
3. Server validates token → connection established.
4. Non-sensitive connection info (host, port, name) stored in app preferences.
5. Token stored in secure storage.

### QR Code Pairing

1. Desktop generates a QR code encoding:
   ```
   miwarp://connect?host={localIp}&port={port}&token={token}
   ```
2. QR code is displayed in Settings > Web Server > Mobile Access.
3. User scans QR code with mobile app camera.
4. Mobile app parses the deep link and connects.
5. Token extracted from URL and stored securely.

### Deep Link Pairing

1. Desktop generates a pairing link:
   ```
   miwarp://connect?host={localIp}&port={port}&token={token}
   ```
2. User opens link on mobile device (e.g., via AirDrop, messaging, or shared note).
3. Mobile app opens, parses the link, and connects.

### Security Considerations for QR/Deep Link

- QR code is displayed only on the desktop screen (not in notifications).
- Token in the QR/link is the same token used for all connections.
- Regenerating the token invalidates ALL existing connections (mobile and browser).
- Users should only scan QR codes from their own desktop.

## Network Security

### Bind Address

| Bind | Behavior | Risk |
|------|----------|------|
| `127.0.0.1` | Localhost only — mobile CANNOT connect | Safest |
| `0.0.0.0` | All interfaces — mobile CAN connect | LAN exposure |

The desktop NEVER automatically changes the bind address. The user must explicitly switch to LAN mode.

### LAN Exposure Warning

When the user enables LAN access (`0.0.0.0`), the desktop shows a warning:

> "Mobile access is enabled. Any device on your local network can connect to MiWarp with this token. Only enable this on trusted networks."

### What the Server Exposes

- `/health` — `{"status": "ok"}` (no auth required, no sensitive data)
- `/ws` — WebSocket endpoint (requires token)
- All other routes — require cookie or token auth

### What the Server Does NOT Do

- No automatic port forwarding
- No UPnP/NAT-PMP
- No public internet exposure
- No cloud relay
- No third-party authentication

## Permission Security

### Mobile Approval

Mobile clients can approve or deny tool permissions. The security rules are:

1. **No auto-approval**: The mobile client NEVER automatically approves permissions.
2. **No default allow**: If the user doesn't respond, the permission times out (server-side).
3. **Dangerous tools**: Tools flagged as `dangerous` in the permission prompt show a more prominent warning.
4. **Deny with message**: Users can provide a reason when denying.
5. **Session-level allow**: "Allow for session" only applies to the current session.

### Permission Flow

```
Desktop CLI requests permission
  → Server sends permission_prompt event to mobile
  → Mobile shows ApprovalCard
  → User taps Allow/Deny
  → Mobile calls respond_permission
  → Server forwards decision to CLI
```

If the mobile client is disconnected when a permission prompt is sent:
- The CLI waits for a timeout (server-configured).
- On reconnect, the mobile client receives the permission_prompt via event replay.

## Data Security

### What Stays on Device

- Connection credentials (host, port, token)
- Cached run history
- Chat messages
- User preferences

### What is Synced

- Real-time events via WebSocket (not persisted by the server beyond `events.jsonl`)
- No data is sent to any cloud service by the mobile app

### Clear All Data

Both iOS and Android provide a "Clear All Connections and Data" option that:
1. Deletes all saved connections
2. Removes all tokens from secure storage
3. Clears cached run data
4. Resets the app to factory state

## Desktop-Side Security

### Token Regeneration

- Regenerating the token disconnects ALL WebSocket clients (mobile and browser).
- Old token becomes invalid immediately.
- New token must be shared with mobile clients (new QR code or manual entry).

### Web Server Status

The desktop exposes `get_web_server_status` which returns:
```json
{
  "enabled": true,
  "running": true,
  "port": 9476,
  "bind": "0.0.0.0",
  "warning": null
}
```

This does NOT include the token. The token is only available via Tauri IPC (`getWebServerToken()`), never over WebSocket.

### CORS

- Non-browser clients (mobile native apps) always pass CORS checks (no Origin header).
- Browser clients are checked against allowed origins.
- This means mobile apps are not restricted by CORS.

## Recommendations for Users

1. **Use LAN-only**: Keep bind as `127.0.0.1` unless actively using mobile access.
2. **Trusted networks**: Only enable mobile access on trusted home/office networks.
3. **Regenerate regularly**: Regenerate the token periodically.
4. **VPN for remote access**: Use a VPN or SSH tunnel for access outside the LAN.
5. **Don't share pairing links**: The pairing link contains the token — treat it as a password.
6. **Monitor connections**: Check the desktop's web server status for unexpected connections.

## Future Enhancements

- **End-to-end encryption**: TLS or Noise protocol between mobile and desktop.
- **Per-device tokens**: Each mobile device gets its own token, revocable individually.
- **Push notifications**: Server-initiated notifications for permission prompts.
- **Cloud relay**: Optional NAT traversal for remote access without VPN.
