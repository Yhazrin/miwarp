This document freezes the types and invariants for Send Transactions,
Permission Transactions, and Recovery state machines in v1.0.9. It is the
**only** place these are defined; any new transaction shape added in code
without a corresponding addition here is a contract violation.

## 0. Inherited invariants from v1.0.9 Phase 1 / 2

The following already exist and MUST NOT be duplicated:

1. **Connection Generation** â€” a monotonically increasing counter on every
   `(run_id, connection)` pair. The frontend must reject any response whose
   `connection_generation` is older than the one it has.
2. **Request Registry** â€” `RequestId â†’ oneshot::Sender<Response>` map with
   capacity cap; the runtime routes responses back to the originating
   caller.
3. **Run Subscription Ownership** â€” exactly one owner per run_id; transfer
   requires explicit handoff.
4. **Bounded Chunk Assembly** â€” chunk decoder with hard size + count caps.
5. **SessionRunConnection** â€” wraps the per-run mailbox; never shared.
6. **SessionRecoveryController** â€” existing recovery entry point; Agent A
   extends, never replaces.
7. **SendCoordinator (Phase 1/2)** â€” `submitting / queued / recovering /
   accepted / failed / cancelled` state machine; already proven in
   `src/lib/chat/use-send-message.ts` and unit tests.
8. **`accepted_client_message_ids` ledger** â€” prevents replay of accepted
   user turns after actor restart.
9. **PermissionCoordinator** â€” single-flight per `(run_id, request_id)`
   with double-click dedupe.
10. **tool_name â†” pending control_request consistency check** â€” backend
    refuses to dispatch a permission response that does not match the
    pending request's tool_name.
11. **`NEVER_ALLOW_TOOLS`** â€” denylist enforced in PermissionCoordinator.
12. **BusEvent cross-platform contract** â€” `models.rs` â†” `types.ts` synced.

## 1. SendTransaction (extends Phase 1/2 SendCoordinator)

```rust
pub struct SendTransaction {
    pub client_message_id: ClientMessageId,   // uuid v7
    pub run_id: RunId,
    pub session_id: SessionId,
    pub generation: ConnectionGeneration,     // current connection generation
    pub state: SendState,

    pub last_transition_at: Instant,

    pub attempts: u32,                        // capped at 3
    pub attachments: Vec<AttachmentRef>,       // dereferenced lazily
    pub idempotency_key: IdempotencyKey,      // == client_message_id
}

pub enum SendState {
    Submitting,   // request typed into coordinator
    Queued,       // waiting for connection or for prior transaction to finish
    Recovering,   // connection lost; flush pending after reconnect
    Accepted,     // backend acked; entry added to accepted_client_message_ids
    Failed(SendError),
    Cancelled,    // user explicitly cancelled before Accepted
}
```

## 2. PermissionTransaction

```rust
pub struct PermissionTransaction {
    pub permission_request_id: PermissionRequestId,
    pub run_id: RunId,
    pub session_id: SessionId,
    pub generation: ConnectionGeneration,
    pub tool_name: ToolName,
    pub tool_input: Value,                    // validated against schema
    pub risk: RiskLevel,                      // see Â§3
    pub state: PermissionState,
    pub created_at: Instant,
    pub decided_at: Option<Instant>,
    pub decision: Option<PermissionDecision>, // AllowOnce | AllowSession | Deny | DenyStopRun
    pub last_error: Option<PermissionError>,  // typed
}

pub enum PermissionState {
    Pending,          // waiting for user
    Deciding,         // user clicked, response in flight
    Decided,          // terminal success
    Stale,            // generation moved on / runtime replaced
    Failed(PermissionError),
    Cancelled,        // run cancelled
}
```

### Invariants

1. **Composite key**: `(run_id, permission_request_id)` is the only valid
   identity. Cross-session requests are dropped at the boundary.
2. **Double-click dedupe**: a second `respond` while in `Deciding` MUST
   resolve to the same in-flight response, not start a new one.
3. **Tool name validation**: `respond_permission(tool_name)` MUST match the
   pending request's `tool_name`; mismatch â†’ `PermissionError::ToolNameMismatch`.
4. **`NEVER_ALLOW_TOOLS`**: any tool in this list, if `AllowSession` is
   requested, is downgraded to `DenyStopRun` and the user is notified via
   `perm_dangerBlocked` i18n key.
5. **Stale handling**: a `respond` whose `generation` is older than the
   current `connection_generation` is dropped with `Stale` and the user sees
   `perm_stale`.
6. **Offline discipline**: a `Decided` state is never emitted from the
   frontend. It is emitted by the backend only after the Runtime has
   acknowledged. If the connection is lost before ack, the state is
   `Deciding` until reconnect, then either `Decided` (if the Runtime
   already saw it) or `Failed(Transport)` (if it didn't).

## 4. RecoveryState (extends SessionRecoveryController)

```rust
pub enum RecoveryState {
    Healthy,           // last op succeeded, no in-flight recovery
    Degraded,          // last op succeeded after retry
    Reconnecting,      // attempting to re-establish connection
    Recovering,        // replaying accepted_client_message_ids
    Recovered,         // recovery completed, generation bumped
    Unrecoverable,     // recovery failed 3 times; user action required
}
```

### Rules

- `Reconnecting` and `Recovering` together MUST NOT exceed 30s. After that,
  the state machine MUST transition to `Unrecoverable`.
- `Recovered` MUST emit a `ConnectionGenerationBumped` event so every
  subscriber can revalidate their own transaction state.
- A `Recovering` session MUST NOT silently drop in-flight
  `SendTransaction` and `PermissionTransaction` entries; they are
  reconciled against `accepted_client_message_ids` and the
  `PermissionCoordinator` ledger.
- `consecutive_failures` is reset only on `Healthy` or `Recovered`. It is
  incremented on every transition into `Reconnecting`.

## 5. Actor lifecycle (extends session_actor.rs)

```rust
pub enum ActorLifecycle {
    Starting,          // spawn() in progress
    Ready,             // first message read
    Crashed { code: Option<i32>, signal: Option<i32>, reason: CrashReason },
    Respawning,        // recovery kicked in
    Stopped,           // explicit stop()
    Disposed,          // map entry removed
}

pub enum CrashReason {
    SpawnFailed,           // binary not found / not executable
    VersionTooOld,         // min_version check failed
    AuthFailed,            // 401 / api key missing
    ProtocolDesynced,      // PROTOCOL_DESYNC_THRESHOLD exceeded
    StdinWriteFailed,      // broken pipe
    StdoutEof,             // process exited without farewell
    StaleGeneration,       // user replaced the session while we were running
    Quarantined,           // too many crashes; in QUARANTINE_DEADLINE
    Unknown,
}
```

### Rules

- Every state change MUST emit a `BusEvent::SessionLifecycle` event.
- `Disposed` is terminal. No more events from this actor.
- A `Respawning` actor MUST inherit `accepted_client_message_ids` from
  the ledger; replaying a non-accepted message after a crash is a bug
  (`RuntimeError::ReplayOfUnaccepted`).
- Three consecutive `Crashed` events with the same `CrashReason` within
  60s MUST escalate to `RuntimeHealth::Unrecoverable`.

## 6. Frontend projection

The frontend consumes these transactions through three composables:

| Composable | Owns |
|---|---|
| `useSendTransaction(runId)` | one `SendTransaction` per submit |
| `usePermissionTransaction(requestId)` | one `PermissionTransaction` per request |
| `useRecoveryState(runId)` | recovery state machine for the current run |

The composables are state-only projections. The single state-of-record
lives in `SendCoordinator`, `PermissionCoordinator`, and
`SessionRecoveryController` respectively. Composables must not write
back directly; they emit intent through the existing handler API.

## 7. Observability requirements

Every transition emits a `DiagnosticEvent` with:

- `category` âˆˆ `SendTx | PermissionTx | Recovery | ActorLifecycle`
- `severity` âˆˆ `Info | Warn | Error`
- `connection_generation`
- `client_message_id` (Send) or `permission_request_id` (Permission) or
  `run_id` (Recovery / Lifecycle)
- `phase` = old state + arrow + new state
- `error_code` (typed) if `severity >= Warn`
- `durationMs` for non-instantaneous transitions

Agent D owns the writers; Agent A owns the call sites. The `last_error`
field on a transaction is mirrored into the diagnostic event with the
sanitizer running (no prompt text, no API keys).

## 8. Forbidden patterns

- Two `SendCoordinator` implementations side by side. There is exactly one.
- A second recovery controller. Agent A extends `SessionRecoveryController`.
- A typed error encoded as a JSON string at the IPC boundary.
- A permission response that bypasses the composite key check.
- A `Recovered` transition that does not bump the connection generation.
- An `Accepted` SendTransaction that is not appended to
  `accepted_client_message_ids`.
- A `Decided` PermissionTransaction emitted from the frontend.
- A `Cancelled` SendTransaction that the UI shows as "Send" still active.
