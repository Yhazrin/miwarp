# v1.0.9 Rollback Checklist

> **Date**: 2026-06-22
> **Status**: Reference document — to be executed only if a v1.0.9
> release is rolled back to v1.0.8 in production.

This document enumerates the steps to revert a v1.0.9 deployment to
v1.0.8 in production. The rollback is destructive in the sense that
any user data created on v1.0.9 is preserved (the storage layer is
forward-compatible), but the v1.0.9-only command surface (e.g.
`runtime_hub_*`) is unavailable until the user re-upgrades.

The rollback assumes v1.0.8 is the immediately-prior release and is
still available on the distribution channel. If v1.0.8 has been
yanked, the rollback target is the v1.0.8 tag on GitHub (downloadable
via the GitHub releases page).

---

## 1. Decision criteria (when to roll back)

A rollback is warranted when one of the following is true in
production:

1. **A critical user-facing flow is broken** (e.g. the app cannot
   create a session, the chat input is unresponsive, the permission
   panel crashes).
2. **A data-loss bug is reported** (e.g. accepted turns are not
   persisted, recovery destroys user state).
3. **A security vulnerability is discovered** that v1.0.8 does not
   have (e.g. a permission bypass that the v1.0.9 trust center was
   supposed to fix but didn't).
4. **A regression in cross-platform parity** (e.g. the iOS app
   cannot connect to the desktop backend).

The decision is made by the release lead (currently the user) based
on the issue tracker. A "non-critical" bug (e.g. UI typo) is NOT a
rollback trigger — it goes into the v1.0.9.1 patch release.

## 2. Pre-rollback validation

Before executing the rollback, the release lead confirms:

1. The bug is reproducible on a clean install of v1.0.9.
2. The bug does NOT exist in v1.0.8 (test on a v1.0.8 install).
3. The v1.0.8 installer is available on the distribution channel
   (GitHub releases, the App Store, the auto-updater feed).

If any of these is false, do not roll back; open a regression
investigation instead.

## 3. Stop the bleeding (the 5-minute triage)

The first 5 minutes of a rollback are about limiting damage, not
fixing root cause.

1. **Pin the v1.0.8 build** in the auto-updater feed. This stops
   new users from getting v1.0.9 via auto-update.
   ```bash
   # On the release tooling repo (Yhazrin/miwarp-releases or
   # similar), pin the latest version to 1.0.8.
   ```
2. **Post a pinned GitHub issue** titled "v1.0.9 rollback in
   progress" with a one-line description of the bug, the
   affected version, and the rollback ETA.
3. **Disable the affected feature** (if possible) via a config flag.
   For example, if the bug is in `runtime_hub_*`, set the
   `MIWARP_DISABLE_RUNTIME_HUB=1` env var on the build server so
   the v1.0.9 binary can be re-released as v1.0.9.1 without
   shipping the broken feature.
4. **Page the on-call engineer** if the rollback is in response to
   a production incident.

## 4. Execute the rollback (the 30-minute procedure)

The rollback is a coordinated release channel action. It does NOT
require a code change in the v1.0.9 binary — the auto-updater
metadata is updated to point at v1.0.8.

### 4.1 Update the auto-updater feed

```bash
# On the release server:
cd /srv/miwarp-releases
git fetch origin
git checkout v1.0.8
./release.sh --channel stable --version 1.0.8
```

This re-publishes the v1.0.8 build as the current "stable" channel
target. The Tauri updater will then offer v1.0.8 to any v1.0.9
user who runs the auto-update.

### 4.2 Yank the v1.0.9 release on GitHub

```bash
gh release delete v1.0.9 --yes
gh release create v1.0.9-rollback --notes "v1.0.9 has been rolled back. Use v1.0.8 instead. See https://github.com/Yhazrin/miwarp/issues/<N> for context."
```

Note: GitHub does not allow editing a release's tag. Deleting and
re-creating is the standard procedure.

### 4.3 Restore the integration branch to v1.0.8

On the integration worktree:

```bash
cd /path/to/integration-worktree
git checkout master
git reset --hard v1.0.8
git push origin master --force-with-lease
```

This is a destructive operation; the v1.0.9 commits on `master`
become orphaned but are recoverable via the reflog for 90 days.

### 4.4 Open a regression issue

```bash
gh issue create --title "v1.0.9 rollback: <one-line description>" \
  --body "$(cat <<'EOF'
## Summary
- v1.0.9 was rolled back to v1.0.8 because <bug>.
- Affected users: <percentage / count>.
- Detection time: <UTC timestamp>.
- Rollback time: <UTC timestamp>.

## Root cause
TBD — opened as a follow-up investigation.

## Next steps
1. Reproduce the bug on a clean install.
2. Identify the regression commit (likely from agent X's branch).
3. Land a fix on a `fix/v1.0.9.1-` branch.
4. Tag v1.0.9.1 with the fix.

## Tracking
- v1.0.9 release: <release URL>
- v1.0.8 fallback: <release URL>
EOF
)"
```

## 5. User-side rollback (the "I need to downgrade NOW" path)

For users who need to roll back BEFORE the auto-updater picks up the
v1.0.8 feed (which can take up to 24 hours for a global rollout):

1. Uninstall v1.0.9 via the OS package manager (Windows: "Apps &
   Features"; macOS: drag-to-trash or `brew uninstall`; Linux: the
   distro package manager).
2. Download v1.0.8 from
   https://github.com/Yhazrin/miwarp/releases/tag/v1.0.8.
3. Install v1.0.8 over the empty slot.

User data is preserved: the storage layer at `~/.miwarp/` is
forward-compatible. v1.0.8 reads the v1.0.9 data without
modification (it ignores the v1.0.9-only keys like
`accepted_client_message_ids`).

## 6. Post-rollback verification (the 1-hour audit)

After the rollback, the release lead confirms:

1. **Auto-updater metadata** points at v1.0.8.
2. **GitHub releases** show v1.0.8 as "Latest", v1.0.9 as "Pre-release"
   or deleted.
3. **Issue tracker** has the regression issue open with a clear
   description and an owner.
4. **CI green** on `master` (now pointing at v1.0.8) — the rollback
   did not break the build.
5. **A new `fix/v1.0.9.1-` branch** exists for the bug fix, branched
   from the v1.0.9 release tag (not from master). The v1.0.9
   release tag is preserved in git even after the GitHub release is
   deleted, so the fix branch can reference it.

## 7. Re-release (the v1.0.9.1 path)

The fix lands on a `fix/v1.0.9.1-<short-desc>` branch. The release
lead runs the v1.0.9 release checklist (`v1.0.9-release-checklist.md`)
on this branch. If the gate is green, the branch is tagged
`v1.0.9.1` and the auto-updater feed is updated to point at it.

The re-release is a MINOR bump (v1.0.9 → v1.0.9.1), not a major
bump — the rollback does not change the contract surface, only the
behavior. Users who skipped v1.0.9 will receive v1.0.9.1 directly
when they next auto-update.

## 8. Lessons-learned (the 1-day review)

After the rollback is complete, the release lead schedules a
60-minute post-mortem with the engineering team. The agenda:

1. What was the bug? (root cause, not just symptom)
2. Why did the gate miss it? (which `arch:check` or test should
   have caught it but didn't?)
3. What is the follow-up contract change? (a new test in
   `e2e/__tests__/` or `src-tauri/tests/`)
4. Is the release checklist still complete? (any v1.0.9-only step
   that needs to be re-checked?)

The post-mortem result is committed to
`docs/architecture/v1.0.9.1-postmortem.md` (created by the release
lead; the template is in `docs/architecture/templates/`).

## 9. ADR for the rollback (the durable record)

The rollback decision is recorded as an ADR in `docs/adr/`:

```bash
cp docs/adr/0007-single-flight-recovery-and-transport-health.md \
   docs/adr/0008-v1.0.9-rollback.md
```

Then edit the new file to describe:

- **Status**: Accepted (the rollback was executed)
- **Context**: what bug triggered the rollback
- **Decision**: rollback to v1.0.8; release v1.0.9.1 with the fix
- **Consequences**: any forward-compatibility concerns, any
  contracts that need to be re-affirmed in the patch release

The ADR is a permanent record; it does NOT need to be reopened
after the fix lands. A separate ADR (e.g. 0009) records the fix.

## Appendix A: Rollback vs. patch — when to use which

| Scenario | Action |
|---|---|
| Critical user-facing flow broken in v1.0.9 | Roll back to v1.0.8 |
| Data-loss bug introduced in v1.0.9 | Roll back + open investigation |
| Security vulnerability in v1.0.9 trust center | Roll back; the fix is a v1.0.9.1 |
| UI bug (e.g. typo, off-by-one) | Patch (v1.0.9.1), no rollback |
| Performance regression in v1.0.9 hub | Patch (v1.0.9.1), no rollback |
| Cross-platform parity break on iOS | Patch (v1.0.9.1), no rollback |
| User wants the v1.0.8 feature X back | Forward-fix; document the removal |

The patch is preferred when the bug is non-critical AND a fix is
identifiable in < 1 day. The rollback is the right call when the
fix is non-trivial OR the user impact is severe.

## Appendix B: Quick-reference commands

```bash
# Pin auto-updater to v1.0.8
./release.sh --channel stable --version 1.0.8

# Yank v1.0.9 on GitHub
gh release delete v1.0.9 --yes

# Restore integration to v1.0.8
git checkout master && git reset --hard v1.0.8

# Open regression issue
gh issue create --title "v1.0.9 rollback: <bug>"

# Re-install v1.0.8 (user-side)
gh release download v1.0.8 --pattern "*.dmg"
# or
gh release download v1.0.8 --pattern "*.exe"
```

## Appendix C: Storage forward-compatibility

The `~/.miwarp/` storage is forward-compatible across v1.0.8 and
v1.0.9:

- v1.0.9-only keys (e.g. `runtime_hub_*` records,
  `accepted_client_message_ids`) are ignored by v1.0.8.
- v1.0.8-only keys (e.g. legacy `SessionLifecycle::Spawned`
  events) are still emitted by v1.0.9 (the new variants are
  additive).
- The schema is documented in
  `docs/architecture/storage-schema.md` (Agent D owns this
  in v1.0.9; until then, the implicit schema is in
  `src-tauri/src/storage/`).

Rollback does NOT require any data migration. The user data
created on v1.0.9 will be visible to v1.0.8 with no
transformation.
