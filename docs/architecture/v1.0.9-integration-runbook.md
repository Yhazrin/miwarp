# v1.0.9 Integration Runbook

> **Date**: 2026-06-22
> **Branch**: integration/v1.0.9-runtime-hub
> **Owner**: Integrator (the user) + Agent H (this runbook)

This is the third v1.0.9 release-quality document. It is a
copy-paste runbook for the integrator — every block of commands
is runnable as-is, and the doc records the EXPECTED output for
each step. If a step's output does not match, the integration is
not ready to merge.

This document is a SUPPLEMENT to:

- `v1.0.9-release-checklist.md` — the 10-step pre-tag gate.
- `v1.0.9-rollback-checklist.md` — the rollback procedure.

The release / rollback checklists explain WHY; this runbook
explains HOW.

## 1. Pre-merge smoke (run on every PR)

```bash
cd /path/to/integration-worktree

# 1. Confirm branch and base commit.
git rev-parse --abbrev-ref HEAD
# Expected: test/v109-release-quality (or whatever agent branch)
git log --oneline -1
# Expected: 5 commits, no `WIP` prefix

# 2. Confirm no forbidden patterns.
git grep -nE "TODO.*v1\.0\.9" || echo "OK: no v1.0.9 TODOs"
git grep -nE "FIXME.*v1\.0\.9" || echo "OK: no v1.0.9 FIXMEs"
# Expected: OK (no matches)
```

## 2. Wave 5 gate (run on the integration branch)

```bash
cd /path/to/integration-worktree

# 3. The full verify pipeline.
npm run verify 2>&1 | tee /tmp/miwarp-verify.log
# Expected: 0 errors, 0 new warnings. Existing warnings (153 lint,

# 4. The strict architecture gate.
npm run arch:check:strict

#   ✓ arch:ios-ws-contract

  --manifest-path src-tauri/Cargo.toml 2>&1 | tail -3
```

## 5. Tag and publish

```bash
cd /path/to/integration-worktree

# 9. Sync version.
npm run version:check
# Expected: all manifests report 1.0.9 (or 1.0.9-rc.1 for RC).
# If not, run `npm run version:sync` and commit.

# 10. Tag the release.
git tag -s v1.0.9 -m "Release v1.0.9: Runtime Hub + trustworthy permissions"
git push origin v1.0.9
# Expected: tag is signed; push succeeds; GitHub release pipeline
# starts.
```

## 6. Post-tag verification (the 24-hour audit)

```bash
# 11. Confirm the release pipeline finished.
gh run list --workflow "release.yml" --limit 1
# Expected: 1 successful run, on the v1.0.9 tag.

# 12. Download the installer and check the version.
gh release download v1.0.9 --pattern "*.dmg" --dir /tmp/miwarp-1.0.9
# macOS: open the DMG, drag to Applications, launch, check
# the version in About.
# Expected: "MiWarp 1.0.9"
```

## 7. Rollback (the 30-minute procedure)

If a regression is reported after the tag:

```bash
# 13. Stop the bleeding.
./release.sh --channel stable --version 1.0.8
gh release delete v1.0.9 --yes
# Expected: auto-updater feed now points at v1.0.8.

# 14. Restore the integration branch.
git checkout master
git reset --hard v1.0.8
git push origin master --force-with-lease
# Expected: master now points at v1.0.8. The v1.0.9 commits are
# recoverable via reflog for 90 days.

# 15. Open a regression issue (see v1.0.9-rollback-checklist.md §4.4).
```

## 8. Re-release (the v1.0.9.1 path)

Once the fix lands on a `fix/v1.0.9.1-<short-desc>` branch:

```bash
# 16. Repeat the Wave 5 gate.
cd /path/to/fix-branch-worktree
git checkout fix/v1.0.9.1-<short-desc>
npm run verify
npm run arch:check:strict
# Expected: 0 errors. The fix-specific tests are added by the
# agent that owns the fix.

# 17. Tag the patch release.
git tag -s v1.0.9.1 -m "Release v1.0.9.1: <fix description>"
git push origin v1.0.9.1
```

## 10. Common pitfalls

1. **Forgetting `npm install`** in a fresh worktree. The
   `npm run verify` step runs `prettier --check` on
   `src/`, which fails if `node_modules/` is empty.
2. **Forgetting `npm run build`** before `cargo test`. The
   Tauri build expects a `build/` dir at the repo root for
   some commands.
3. **Skipping `cargo fmt`** after editing a Rust file. The
   pre-commit hook runs `cargo fmt --check`; if it fails,
   the commit is rejected.
4. **Modifying files outside your ownership**. The
   `arch:direction` + `arch:layers` + `arch:cycle` gates
   enforce the dependency direction. A new import that
   crosses a layer boundary fails the gate.
5. **Adding a `runtime_hub_*` command** without an ADR. The
   `arch:runtime-contract` test asserts exactly 4
   `runtime_hub_*` commands; a 5th without a spec change
   fails the gate.
6. **Yanking the integration branch on push**. The
   `arch:budget:diff` gate compares against the merge
   base; if you `git push --force`, the base changes and
   the diff range shifts.
7. **Forgetting the `frontend build` step** in CI. The
   `tauri::generate_handler!` macro needs `$CARGO_MANIFEST_DIR/../build`
   to exist (a stub is fine for `cargo check`).

## 11. CI command reference (the JSON output)

The CI uses `arch:check`, which is a chain of 8 sub-gates.
The output looks like:

```
✓ arch:direction: 0 violations
✓ arch:layers: 0 violations
✓ arch:cycle: 0 violations
✓ arch:budget: 0 violations
✓ arch:tauri-contract: 0 violations
✓ arch:ios-ws-contract: 0 violations
✓ arch:cross-platform-bus: 0 violations
✓ arch:runtime-contract: 9 passed
```

If any line starts with `✗`, the gate is red and the
integration is not ready to tag. The line after `✗` shows
the file + line + violation description.

is allowed to NOT add business features — it's the test
scaffolding.

## 13. What the integrator does NOT touch

The integrator DOES NOT modify any of these files unless
explicitly required by the release checklist:

- `src-tauri/src/agent/**` (Agent A's sole ownership)
- `src-tauri/src/agent/protocol/**` (Agent B's strangler)
- `src-tauri/src/observability/**` (Agent D's new module)
- `src/lib/observability/**` (Agent D's frontend)
- `src/lib/workbench/**` (Agent E's new module)
- `src-tauri/src/diagnostics/**` (Agent D's new module)
- `apps/ios/MiWarpMobile/**` (Agent G's iOS)
- `apps/android/**` (Agent G's Android)
- `src/lib/runtime/**` (Agent B's frontend runtime hub)
- `scripts/architecture/**` (Agent H's arch scripts)

The integrator's role is the merge, the version sync, the
tag, the release notes, and the rollback if needed.
