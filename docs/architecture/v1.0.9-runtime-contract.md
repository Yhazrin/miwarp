Anything that contradicts the types in this document is a bug.

## 1. Three concepts, deliberately separated

MiWarp v1.0.9 enforces the distinction between three things that earlier
versions conflated:

| Concept | Question it answers | Example | Persisted? |
|---|---|---|---|
| **Provider** | Where do the inference calls go? | `anthropic-direct`, `azure-relay`, `local-ollama`, `internal-proxy` | per-user settings |
| **Runtime** | What process executes the user turn and owns the protocol? | `claude-code`, `codex`, `mimo-code`, `ssh-remote-linux` | per-project default + per-session override |
| **Model** | What model does the Runtime call? | `claude-opus-4.7`, `gpt-5.1`, `mimo-pro-1` | per-session, switchable mid-run via provider model list |

A single Provider can host multiple Runtimes. A single Runtime can call
multiple Models (provider-side). Mixing these in a single struct was the
root cause of the v1.0.8 condition where "model missing" and "binary missing"
both produced the same `RuntimeNotReady` error.

## 2. RuntimeDescriptor (immutable identity)

```rust
pub struct RuntimeDescriptor {
    pub id: RuntimeId,                  // "claude-code", "codex", "mimo-code", "ssh:<host>"
    pub display_name: String,            // localized via i18n key, not struct field
    pub kind: AgentRuntimeKind,          // existing enum
    pub protocol: RuntimeProtocolKind,   // existing enum
    pub version: RuntimeVersion,         // semver parsed from --version, or "unknown"
    pub source: RuntimeSource,           // LocalBinary | ManagedBundle | SshRemote | HttpRelay
    pub capabilities: RuntimeCapabilities, // see §3
    pub launch: RuntimeLaunchSpec,       // see §4
}
```

`RuntimeDescriptor` is the **only** type that the frontend may use to render
"which Runtime am I talking to". The string `id` is stable; everything else
can change across launches.

## 3. RuntimeCapabilities (what the Runtime can do)

```rust
pub struct RuntimeCapabilities {
    pub supports_streaming: bool,        // SSE / pipe / chunked
    pub supports_resume: bool,           // --resume / fork
    pub supports_fork: bool,             // start a new run from a prior run_id
    pub supports_permission_requests: bool,
    pub supports_tool_calls: bool,
    pub supports_usage: bool,            // tokens / cost
    pub supports_thinking: bool,
    pub supports_attachments: bool,      // images, pdf, audio
    pub supports_images: bool,
    pub supports_mcp: bool,
    pub supports_skills: bool,
    pub supports_remote_execution: bool, // ssh or relay
    pub supports_structured_events: bool,// reliable event stream with bus-event mapping
}
```

**Rule**: if a capability is `false`, the UI must hide the action before
the action is even attempted. Hiding is the only valid degradation. If a
Runtime claims `supports_streaming = false` but the user clicks an action
that requires streaming, the typed error is `CapabilityUnsupported { cap,
runtime_id }`, never a silent fallback to a different Runtime.

## 4. RuntimeLaunchSpec (how to start it)

```rust
pub struct RuntimeLaunchSpec {
    pub binary: String,                  // resolved absolute path
    pub base_args: Vec<String>,
    pub env: BTreeMap<String, String>,
    pub cwd_policy: CwdPolicy,           // ProjectRoot | UserHome | Custom(PathBuf)
    pub spawn_timeout: Duration,         // must be < 30s
    pub health_check: HealthCheckSpec,   // see §5
}
```

The frontend never reads `binary` or `env` for display. It only reads
`RuntimeHealth.last_diagnosis` and the i18n-rendered `display_name`.

## 5. RuntimeHealth (current observable state)

```rust
pub struct RuntimeHealth {
    pub runtime_id: RuntimeId,
    pub state: RuntimeHealthState,       // see below
    pub last_diagnosis: Option<RuntimeDiagnosis>,
    pub last_healthy_at: Option<Instant>,
    pub consecutive_failures: u32,
    pub connection_generation: u64,      // increments on every reconnect
    pub last_error: Option<RuntimeError>, // typed, not string
}

pub enum RuntimeHealthState {
    Unknown,         // not probed yet
    Detecting,       // running binary --version
    Ready,           // last launch succeeded
    Degraded,        // launched but partial failure (e.g. missing tool)
    Reconnecting,    // active re-launch in flight
    Recovering,      // recovery state machine running (delegates to Agent A)
    Unrecoverable,   // consecutive_failures >= 3
}
```

`Unrecoverable` requires explicit user action: "edit config" or "switch
runtime". The UI must surface this; it must not be auto-retried.

## 6. RuntimeDiagnosis (typed, user-actionable)

```rust
pub struct RuntimeDiagnosis {
    pub category: DiagnosisCategory,     // BinaryNotFound | VersionTooOld | AuthMissing | ...
    pub severity: DiagnosisSeverity,     // Info | Warn | Error | Fatal
    pub title: String,                   // i18n key
    pub body: String,                    // i18n key with placeholders
    pub fix_action: Option<FixAction>,   // OpenSettings | InstallGuide | Retry | SwitchRuntime
    pub retryable: bool,
}
```

All UI messages go through i18n. The Rust side carries only keys +
interpolation variables; the frontend resolves them. **This rule already
applies to existing permission-mode i18n keys and is extended here.**

## 7. RuntimeLaunchSpec::HealthCheckSpec

```rust
pub struct HealthCheckSpec {
    pub probe_args: Vec<String>,         // e.g. ["--version"]
    pub probe_timeout: Duration,         // default 5s
    pub min_version: Option<RuntimeVersion>, // e.g. >= 2.0.0 for claude-code
    pub auth_check: AuthCheckSpec,       // probe env or login state
}
```

The integration worktree must include a Vitest unit test that asserts
`probe_args` for `claude-code` is `["--version"]` and that
`min_version` is enforced; downgrading the contract test must fail.

## 8. Transport boundary

The `RuntimeHub` (Agent B) lives in `src-tauri/src/agent/hub.rs` and exposes
exactly four Tauri commands (Agent H's contract test must verify only these
four exist):

| Command | Direction | Returns |
|---|---|---|
| `runtime_hub_list` | frontend → backend | `Vec<RuntimeDescriptor>` (ordered by `display_name`) |
| `runtime_hub_health` | frontend → backend | `HashMap<RuntimeId, RuntimeHealth>` |
| `runtime_hub_diagnose` | frontend → backend | `RuntimeDiagnosis` for a specific id |
| `runtime_hub_set_default` | frontend → backend | `Result<()>` |

No other Runtime-mutating command may be added without an ADR. This stops
"Runtime Hub" from becoming a dumping ground for one-off CLI launches.

## 9. Capability flags → UI degradation map

| Capability | If false, hide |
|---|---|
| `supports_streaming` | Live tail, progress bar; show a "computing..." spinner instead |
| `supports_resume` | "Resume session" button |
| `supports_permission_requests` | Permission trust center collapses to read-only audit |
| `supports_tool_calls` | Tool result previews, workbench file diff linking |
| `supports_usage` | Usage / cost panel |
| `supports_thinking` | "Show reasoning" toggle |
| `supports_attachments` | Image / file attachment buttons |
| `supports_images` | Image preview in tool results |
| `supports_mcp` | MCP marketplace and live MCP servers view |
| `supports_skills` | Community skills browse / install |
| `supports_remote_execution` | "Local-only" badge on every run |
| `supports_structured_events` | BusEvent timeline (fallback to raw log tail) |

This table is normative. Agent C/E/F must consume it; the `useRuntimeCapabilities()`
composable (Agent B) is the single source.

## 10. Forbidden patterns (enforced by review)

- A new `if runtime == "claude" { ... } else if runtime == "codex" { ... }`
  block in any UI/store file. Use `RuntimeCapabilities` instead.
- A Runtime that silently downgrades streaming to polling. Return
  `CapabilityUnsupported` and let the user choose.
- A capability flag added without updating §9 and the contract test.
- A typed error that erases to a string at the IPC boundary.

## 11. Migration path (strangler)

`claude_protocol.rs` (3409 lines) is the largest protocol surface. Agent B
must migrate it incrementally:

1. Extract `RuntimeMessage` enum (one variant per protocol event family)
   without behavior change.
2. Add `ProtocolAdapter` trait (parse / serialize / capability report).
3. Implement `ClaudeProtocolAdapter` first, then `CodexProtocolAdapter`,
   then `MimoProtocolAdapter`. Each adapter unit-tested in isolation.
4. Replace call sites in `session_actor.rs` via the trait one event family
   at a time. After each event family, `arch:budget` and the test suite
   must still pass.
5. `claude_protocol.rs` becomes the Claude adapter plus a `mod.rs` that
   re-exports the trait. No other module may still import from the old
   `claude_protocol::*` directly once the migration is complete.

The strangler MUST ship behind the existing CLI path; turning it off is a
single feature flag. Agent H owns the test matrix that proves
`turn off → turn on` is behavior-preserving.
