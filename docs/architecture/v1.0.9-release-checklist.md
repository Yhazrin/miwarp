# v1.0.9 Release Checklist

> **Date**: 2026-06-22
> **Branch**: integration/v1.0.9-runtime-hub
> **Status**: TO BE COMPLETED before tagging v1.0.9

This checklist enumerates the verification steps the integrator MUST run
before tagging v1.0.9. Each step is a single command (or short sequence)
that produces a pass/fail signal. The gate is "all green" — any red row
blocks the release.

This is the release-quality contract for v1.0.9. It is owned by Agent H
(`test/v109-release-quality`) but the integrator (the user) is the
executor.

---

## 1. Pre-tag validation (Wave 5 gate)

The full CI gate is `npm run verify`. Run it once and confirm 0 errors:

```bash
cd /path/to/integration-worktree
npm run verify
```

The expected output (in order, all must be clean):

- `npm run check:root` — no pollution in repo root.
- `npm run lint` — 0 errors. Warnings are allowed only if pre-existing.
- `npm run format:check` — 0 unformatted files.
- `npm run check` — svelte-check, 0 errors / 0 warnings.
- `npm run i18n:check` — every new i18n key in both `en.json` and
  `zh-CN.json` with matching placeholders.
- `npm test` — all vitest tests pass.
- `npm run build` — Vite production build succeeds.
- `npm run rust:check` — `cargo fmt --check` + `cargo clippy -- -D warnings`.

If any step is red, the integration is not ready to tag. The integrator
opens a regression issue and rolls back the offending commit (see
`v1.0.9-rollback-checklist.md` for the revert procedure).

## 2. Architecture gate (the v1.0.9-specific assertions)

```bash
npm run arch:check
```

This runs all six baseline checks (direction, layers, cycle, budget,
tauri-contract, ios-ws-contract) PLUS the v1.0.9-specific additions:

- `arch:cross-platform-bus` — runtime_hub_* / diagnostics_* cross-platform
  reachability; Runtime*/Diagnostic* BusEvent variants in sync.
- `arch:runtime-contract` — the spec doc lists exactly 4 runtime_hub
  commands, 12 capability flags, and full §3/§9/§4 type coverage.

For strict mode (recommended for release tags):

```bash
npm run arch:check:strict
```

This adds `arch:budget:diff` — the per-commit file-size diff budget
gate. The integrator runs this against the integration HEAD to confirm
no new/modified file in the v1.0.9 diff range exceeds the fail
threshold.

## 3. Contract test suite (forward-looking enforcement)

The contract tests in `scripts/architecture/__tests__/` and
`src-tauri/tests/` activate as Agent A/B/C/D land their work. The
expected state at tag time:

- `runtime-contract.test.ts` — spec doc passes (always green). The
  deferred assertions for `useRuntimeCapabilities` and the 4-command
  hub ARE enforced at tag time.
- `send_fault_injection.rs` — 15 tests covering the SendTransaction
  state machine. Must be green.
- `permission_fault_injection.rs` — 13 tests covering the
  PermissionTransaction wire format and NEVER_ALLOW_TOOLS. Must be
  green.
- `runtime_contract_types.rs` — 6 tests covering the BusEvent wire
  format for the runtime/diagnostic namespace. Must be green.
- `bus-contract.test.ts` (pre-existing) — every Rust BusEvent variant
  has a matching frontend type literal. Must be green.
- `redaction-manifest-contract.test.ts` (Agent D) — the diagnostics
  redaction manifest declares all required fields. Must be green.

If any of these is red, the integration is not ready. Open a regression
issue.

## 4. Cross-platform smoke test

```bash
npm run e2e:golden
```

This runs the Playwright e2e golden-path spec on the dev build. The
expected outcome:

- 1 test passes (the "end-to-end smoke" test).
- 0 tests skipped, 0 tests failed.

A skip is OK ONLY if Playwright isn't installed. If Playwright IS
installed and the test is skipped, the spec is wrong — open a regression.

This step requires a running Tauri dev server (`npm run tauri dev`) and
Playwright (`npx playwright install --with-deps`). On CI, the lane
explicitly opts in.

## 5. Manual smoke (the "does it actually work?" gate)

The integrator (or a designated user) runs the following manual steps
on a clean machine. Each step is a yes/no question — any "no" blocks
the release.

1. Install the v1.0.9 build via the local installer.
2. Launch the app. Does the splash screen appear within 3s?
3. Open a project folder. Does the file tree render?
4. Create a new session. Does the chat input appear?
5. Send a message. Does the response arrive within 60s?
6. Trigger a permission prompt (e.g. ask the CLI to read a file in
   `~/`). Does the trust center show a clear prompt with Allow / Deny
   / Always Allow buttons?
7. Approve the permission. Does the tool result render inline?
8. Switch sessions. Does the previous session's state persist?
9. Open Settings → Diagnostics. Does the ring buffer show recent
   events?
10. Click "Export diagnostics". Does a `.zip` download with
    `manifest.json` + `events.jsonl` appear in the OS download folder?
11. Restart the app. Does the previous session restore correctly?
12. Disconnect the network (turn off Wi-Fi). Send a message. Does the
    UI show "queued" / "reconnecting" / "offline" state without
    crashing?
13. Reconnect. Does the queued message flush in order?
14. Open the same project on the iOS app (if v1.0.9 ships the mobile
    build). Does the session list sync? Can the user send a message
    from the iOS app and see it on the desktop?

Any failure opens a regression issue. The rollback procedure is
described in `v1.0.9-rollback-checklist.md`.

## 6. Cross-platform capability matrix check

Read `docs/architecture/cross-platform-capability-matrix.md` and confirm
the v1.0.9 column is populated. Every platform (Desktop, Browser WS,
iOS, Android) must have a row in the matrix. Any cell that says
"not supported" requires a known-categorisation note (similar to
the `_full_reload` precedent).

## 7. CHANGELOG and release notes

```bash
npm run release:notes > docs/release-notes/v1.0.9.md
```

Read the generated notes. They should describe:
- The new Runtime Hub / Provider / Model separation.
- The new permission trust center.
- The recovery state machine.
- Any breaking changes (e.g. removed CLI flags, renamed commands).
- Migration steps for users on v1.0.8.

Append a top-level "Known limitations" section listing anything that
is documented but not yet implemented (per the v1.0.9-wave0-baseline.md
"Open questions for user" section).

## 8. Version sync

```bash
npm run version:check
```

The package.json, tauri.conf.json, and src-tauri/Cargo.toml MUST all
report `1.0.9` (or `1.0.9-rc.1` for a release candidate). If not:

```bash
npm run version:sync
```

This bumps the version in all three files. The integrator commits the
sync as a separate `chore: bump version to 1.0.9` commit.

## 9. Tag and publish

```bash
git tag -s v1.0.9 -m "Release v1.0.9: Runtime Hub + trustworthy permissions"
git push origin v1.0.9
```

The signed tag is what triggers the release pipeline. The integrator
confirms the GitHub release is published with the auto-generated notes.

## 10. Post-tag verification

After the release is published, run a smoke install on a clean
machine (a friend / colleague / CI runner). Confirm the installer
downloads, the app launches, and the gold-path flows work. Report
any issues to the maintainer.

## Appendix B: Wave 1/2/3/4 summary (the contract surface that must hold)

For v1.0.9 to release, these contracts MUST be honored:

1. **Runtime Hub separation** — Provider / Runtime / Model are three
   distinct concepts. UI degradation map is in `runtime-contract.md` §9.
2. **Permission transactions** — typed errors, NEVER_ALLOW_TOOLS, double-click
   dedupe, stale generation, transport retry. See `transaction-contracts.md` §2.
3. **Recovery state machine** — Reconnecting / Recovering / Recovered /
   Unrecoverable transitions; generation bump on Recovered; 30s total
   recovery window. See `transaction-contracts.md` §4.
4. **Cross-platform parity** — runtime_hub_* and diagnostics_* commands
   are reachable from Tauri AND iOS WS. The `arch:cross-platform-bus` gate
   enforces this.
5. **BusEvent sync** — every Rust variant has a matching frontend type.
   The general bus-contract test enforces this.
6. **Diagnostics redaction** — the manifest declares prompt / api_key /
   authorization. The `e2e/__tests__/redaction-manifest-contract.test.ts`
   enforces this once Agent D lands.

If any of these is broken, the release is blocked.
