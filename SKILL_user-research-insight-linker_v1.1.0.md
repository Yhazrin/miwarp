# 用研洞察拉通 Skill

## 1. 技能定位

本 Skill 用于读取用研报告、用户反馈、访谈纪要或调研材料，提炼用户洞察，并关联已有产品定义、需求、Idea 和产品 Gap。

它不是普通用研摘要工具。它重点回答：
```
用户到底有什么问题？
当前产品有没有覆盖？
这个问题对应哪个需求或 Idea？
是否存在产品 Gap？
下一步应该验证、观察还是进入需求流程？
```

## 2. 输入质量校验

在开始分析前，必须校验输入数据质量：

| 校验项 | 要求 | 缺失时处理 |
|---|---|---|
| 用户原话 | 输入必须包含至少 3 条用户原话 | 降级提示：「输入缺乏用户原话，分析结果可信度较低，建议补充原始访谈/反馈数据」 |
| 提及频次 | 每条洞察需标注提及人数和总样本数 | 降级提示：「输入缺乏频次数据，无法判断证据强度，建议补充提及统计」 |
| 分群数据 | 至少按 1 个维度分群（如新老用户、使用场景等） | 非必须，但缺失会影响覆盖率分析精度 |

**降级规则**：
- 输入只有结构性分析（无原始数据）→ 输出中所有证据强度降为 `LOW`，并附加质量降级警告
- 输入包含部分原始数据 → 仅对有原始数据的洞察做完整分析，其余标记「数据不足」

# 操作说明

## 操作一：用研拉通（默认）

- **功能**：从用研材料提取洞察，关联产品定义和 Idea
- **输入**：用研文档链接 / 用户反馈文本 / 访谈纪要 / 调研报告
- **输出**：用研洞察拉通结果（主题分析、证据账本、用户问题、产品 Gap、关联需求、关联 Idea）

### 触发场景

- 用户要求整理用研材料
- 用户要求从反馈中找产品机会
- 用户要求找产品 Gap
- 用户要求把用研洞察关联需求或 Idea
- 新用研报告生成后自动入库

### 工作流程

```
读取用研材料
→ 校验输入质量（是否包含用户原话、提及频次、分群数据）
→ 主题分析（将洞察聚类为 3-5 个核心主题）
→ 提取用户问题
→ 提取用户原话或证据
→ 构建证据账本（每条洞察附带 2+ 条用研证据）
→ 判断证据强度（HIGH / MEDIUM / LOW）
→ 检索已有产品定义
→ 产品定义覆盖率分析（覆盖/部分覆盖/未覆盖/过度覆盖）
→ 检索已有需求和 Idea
→ 判断当前产品是否覆盖
→ 标记产品 Gap
→ 生成优先级矩阵（用户问题 × 产品影响）
→ 生成洞察到需求/Idea 映射
→ 建议验证动作
→ 写入 ResearchInsight
```

### 主题分析

在关联产品定义之前，先对输入洞察做主题聚类：

1. 从所有用户问题中提取 3-5 个核心主题
2. 每个主题包含多个相关的用户问题
3. 每个主题附带代表性用户原话和提及频次

**主题聚类原则**：
- 同一主题下的用户问题应指向相同的产品能力或体验环节
- 优先按功能模块聚类，其次按用户旅程阶段聚类
- 避免主题过大（>8 个问题）或过小（仅 1 个问题）

### 证据强度体系

| 强度 | 含义 | 判定标准 |
|------|------|------|
| HIGH | 高可信度 | 多人提及 + 数据支撑（占比 ≥ 30% 或 ≥ 5 人） |
| MEDIUM | 中可信度 | 2-3 人提及，无数据佐证 |
| LOW | 低可信度 | 单人提及，无其他来源交叉验证 |

### 证据账本规则

每条用户问题 / Gap 必须附带 **2+ 条** 用研证据，格式为：

```
[来源文档名] 用户原话 "xxx" (N/M 提及)
```

- N = 提及该问题的用户数
- M = 该主题下的受访用户总数
- 如果某条洞察无法提供 2 条证据，标记为 `LOW` 且在输出中注明「证据不足，需补充验证」

### 产品定义覆盖率分析

对每个用户问题，检查当前产品定义是否已覆盖：

| 覆盖状态 | 含义 | 处理建议 |
|---|---|---|
| 覆盖 | 产品定义已完整覆盖该用户问题 | 无需新增需求，可能需要验证实现效果 |
| 部分覆盖 | 产品定义覆盖了部分场景，但存在遗漏 | 补充需求细节或扩展场景覆盖 |
| 未覆盖 | 产品定义未涉及该用户问题 | 标记为 Gap，建议新增需求或 Idea |
| 过度覆盖 | 产品定义投入过多资源在该问题上，但用户实际提及频次低 | 建议评估资源分配合理性 |

### 优先级矩阵

| 用户问题 | 提及频次 | 产品影响 | 优先级 |
|---|---|---|---|
| 问题描述 | 高/中/低 | 高/中/低 | P0/P1/P2/P3 |

**优先级计算规则**：

|  | 高频提及 | 低频提及 |
|---|---|---|
| **高产品影响** | P0（立即处理） | P1（尽快处理） |
| **低产品影响** | P2（计划处理） | P3（持续观察） |

### 状态规则

| 状态 | 说明 |
|------|------|
| proposed | 单个用户反馈 |
| under_discussion | 多用户重复反馈 |
| confirmed | 已被正式确认的产品 Gap |
| idea | 可转为机会点 |
| pending_confirmation | 证据不足 |

# 降级处理

## 用研材料读取失败

未能读取用研材料。

可能原因：
1. 文档链接无效
2. 文档已被删除
3. 无权限访问

建议：
1. 检查文档链接是否正确
2. 或直接将用研内容粘贴到对话中

## 输入数据质量不足

输入缺乏用户原话或提及频次数据。

建议：
1. 补充原始访谈/反馈数据（含用户原话）
2. 补充提及频次统计
3. 或接受降级分析（所有证据强度标记为 LOW）

## 证据不足

用户反馈证据不足，无法判断是否为真实需求。

建议：
1. 收集更多用户反馈
2. 或标记为"待观察"

## 关联失败

未能找到与用户问题相关的产品定义或 Idea。

建议：
1. 确认产品是否已覆盖该需求
2. 或建议新增 Idea

## 部分内容处理失败

已成功处理 {成功数}/{总数} 条用户反馈。

失败的部分：
- {失败项}：{失败原因}

建议：查看已成功处理的结果，手动补充失败的部分。

# 管道衔接：上游输入模板

当上游是「用研逐字稿分析」skill 时，推荐以下输入格式：

```yaml
source:
  document_name: "用研报告标题"
  document_url: "飞书文档链接"
  date: "YYYY-MM-DD"
  user_type: "目标用户类型"
  sample_size: 12

insights:
  - user_problem: "用户问题简述"
    original_quotes:
      - quote: "用户原话"
        speaker_id: "匿名编号"
        frequency: 3    # 该问题被提及的用户数
        total_users: 12  # 该主题受访用户总数
    segments:          # 分群数据（可选）
      - group: "新用户"
        frequency: 2
      - group: "老用户"
        frequency: 1
    related_feature: "相关功能模块"  # 可选
```

# 验收标准

- 能校验输入质量并给出降级提示
- 能将洞察聚类为 3-5 个核心主题
- 能构建证据账本，每条 Gap 附带 2+ 条证据
- 能区分原话和洞察
- 能判断证据强度（HIGH / MEDIUM / LOW）
- 能分析产品定义覆盖率
- 能生成优先级矩阵
- 能关联已有需求和 Idea
- 能识别产品 Gap
- 不把单点反馈升级为正式需求
- 不输出敏感用户信息
