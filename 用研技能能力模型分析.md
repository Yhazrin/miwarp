# 用研技能深度分析报告

> 分析时间: 2026-07-17
> 分析对象: 两个用研技能的完整能力模型 + 与用研包实际文档的差异

---

## 一、技能 A：用研逐字稿分析与洞察报告

### 能力模型

```yaml
技能名: user-research-skill-matrix / 用研逐字稿分析与洞察报告
版本: 2.3.0-beta.1
核心定位: 从用研逐字稿/座谈会记录中提取证据编码→主题分析→洞察卡→决策建议，输出结构化洞察报告

输入要求:
  必需输入:
    - 逐字稿（座谈会/访谈完整转写）或访谈记录
    - 调研目标/验证问题（来自SOP/OnePage/物料清单，若无则必须询问用户）
  可选输入:
    - SOP文档（提取调研目标、验证问题、样本设计）
    - OnePage（产品背景、核心卖点、目标用户）
    - 物料清单/竞品矩阵（功能对比、竞品定位、价格带）
    - 问卷数据/KANO分析（定量验证、功能分类）
    - 座谈会大纲（测试维度、功能列表）
    - 会议纪要/AI生成纪要（补充参考）
    - 招募条件/筛选结果（样本构成）
  输入格式: 飞书文档链接 / 粘贴文本 / 文件

处理流程:
  - Step 1: 预检 — 识别输入材料（8类材料按优先级排序）→ 提取调研目标 → 识别用研类型（立项前/体验测试/内测上市后/竞品分析）→ 关联结构化数据（竞品矩阵、KANO、问卷）→ 读取座谈会大纲功能列表
  - Step 2: 证据编码 — 从逐字稿逐句提取 OBSERVATION/VERBATIM/INTERPRETATION/INSIGHT/RECOMMENDATION → In Vivo Coding → 一码一事 → 保留上下文 → 区分群体 → 检查易遗漏信号（交互细节/意外好评/未满足场景/意见分歧/自发提及/情绪强度/行为矛盾）
  - Step 3: 主题分析 — 聚类编码 → "当…由于…导致…"结构 → 频次×强度矩阵 → 定量+定性交叉验证 → 竞品数据整合 → 分群分析（用户类型/场景/体型/产品版本/技术方案） → 反例检查
  - Step 4: 洞察提炼 — 只报告产品团队不知道的、可行动的 → 三层深度（表层/中层/深层） → 从洞察到行动（决策影响/建议行动/验证方法/关联数据）

输出格式:
  一级标题:
    - 一、研究预检
    - 二、核心发现速览（P0必备/P1重要/P2增值/关键数字）
    - 三、证据账本
    - 四、主题分析
    - 五、洞察卡
    - 六、决策记录
    - 七、反例与局限
    - 八、下一步建议
  二级标题:
    - 调研目标 / 已知vs未知 / 用研类型与方法 / 结构化数据索引
    - 编号证据条目（含原文引用和置信度）
    - 结构化主题 / 频次强度置信度 / 交叉验证 / 竞品整合 / 分群 / 反例
    - 可行动洞察（含证据/决策影响/建议行动/风险）
    - 决策条目（含建议行动/验证方法/状态）
  关键模块:
    - 证据账本（YAML格式，含evidence_id/source/participant/type/content/scene/confidence）
    - 洞察卡（至少中层深度：要什么+为什么）
    - 核心发现速览（P0/P1/P2分级）
    - 决策记录
    - 质量自检清单（12项）
  表格/评分/引用: 是（大量表格、置信度评分HIGH/MEDIUM/LOW、原文引用VERBATIM）

能力边界:
  能做什么:
    - 从逐字稿提取证据编码（OBSERVATION/VERBATIM/INTERPRETATION/INSIGHT/RECOMMENDATION）
    - 结构化主题分析（频次×强度矩阵、分群、反例检查）
    - 洞察提炼（三层深度：表层/中层/深层机制）
    - 定量+定性交叉验证（逐字稿 vs 问卷/KANO数据）
    - 竞品数据整合（功能对比、价格带、差异化分析）
    - 输出可行动的决策建议
    - 质量自检（12项清单）
  不能做什么:
    - 不做研究设计和访谈提纲生成
    - 不做招募和问卷发放
    - 不做访谈执行和主持
    - 不是问卷模板合集或访谈问题大全
    - 不关联已有产品定义、需求、Idea（这是insight-linker的职责）
    - 不判断产品 Gap
    - 不生成用研需求文档

与用研包5部分的映射:
  OnePage: 不能生成。可读取OnePage作为输入背景，但不输出OnePage。
  用研需求文档: 不能生成。可读取SOP/调研目标，但不输出用研需求文档。
  用研架构设计: 不能生成。不涉及研究设计。
  逐字稿分析: ✅ 能生成，且是核心能力。输出证据账本+主题分析+洞察卡。
  最终报告: ✅ 能生成，输出8节结构化洞察报告（预检→速览→证据→主题→洞察→决策→反例→下一步）。
```

## 二、技能 B：用研洞察拉通

### 能力模型

```yaml
技能名: user-research-insight-linker / 用研洞察拉通
版本: 1.0.0-beta.1
核心定位: 从用研材料提炼用户洞察，关联已有产品定义、需求、Idea 和产品 Gap，回答"用户有什么问题→当前产品有没有覆盖→是否存在 Gap"

输入要求:
  必需输入:
    - 用研文档链接 / 用户反馈文本 / 访谈纪要 / 调研报告
  可选输入:
    - 无显式可选输入（依赖内部知识库的ProductFact表）
  输入格式: 飞书文档链接 / 粘贴文本

处理流程:
  - Step 1: 读取用研材料（feishu_fetch_doc）
  - Step 2: 提取用户问题
  - Step 3: 提取用户原话或证据
  - Step 4: 判断证据强度（弱/中/强）
  - Step 5: 检索已有产品定义（knowledge.search → ProductFact表）
  - Step 6: 检索已有需求和 Idea
  - Step 7: 判断当前产品是否覆盖
  - Step 8: 标记产品 Gap
  - Step 9: 生成洞察到需求/Idea 映射
  - Step 10: 建议验证动作
  - Step 11: 写入 ResearchInsight

输出格式:
  一级标题:
    - 一、用研来源
    - 二、核心用户问题
    - 三、产品 Gap
    - 四、关联需求
    - 五、关联 Idea
    - 六、建议下一步
  二级标题: 无（一级标题下直接是表格）
  关键模块:
    - 核心用户问题表（用户问题/证据/证据强度/来源）
    - 产品Gap表（Gap/当前产品覆盖情况/影响用户/建议动作）
    - 关联需求表（用户问题/已有关联需求/当前状态）
    - 关联Idea表（用户问题/关联Idea/是否建议新增）
    - 建议下一步表（动作/原因/优先级）
  表格/评分/引用: 是（表格为主，证据强度弱/中/强评分）

能力边界:
  能做什么:
    - 从用研材料提取用户问题
    - 判断证据强度（弱/中/强）
    - 关联已有产品定义（从ProductFact知识库检索）
    - 关联已有需求和 Idea
    - 识别产品 Gap（用户问题 vs 当前产品覆盖）
    - 建议下一步动作（验证/观察/进入需求流程）
    - 写入ResearchInsight（持久化）
  不能做什么:
    - 不做逐字稿深度编码（OBSERVATION/VERBATIM级别）
    - 不做主题分析（频次×强度矩阵、分群、反例检查）
    - 不做定量+定性交叉验证
    - 不做竞品数据整合
    - 不做洞察三层深度分析（表层/中层/深层机制）
    - 不把单点反馈升级为正式需求
    - 不输出未脱敏用户隐私
    - 不负责用研报告准确性验证

与用研包5部分的映射:
  OnePage: 不能生成。可读取OnePage作为产品定义参考，但不输出。
  用研需求文档: 不能生成。
  用研架构设计: 不能生成。
  逐字稿分析: ❌ 不能生成。不支持证据编码、主题分析、洞察卡等深度分析。
  最终报告: ⚠️ 部分能生成。输出"用研洞察拉通结果"（6节），但结构较简单，缺少证据账本、主题分析、反例检查、决策记录等模块。
```

## 四、两个技能对比总结

| 维度 | 用研逐字稿分析与洞察报告 | 用研洞察拉通 |
|------|------------------------|-------------|
| **定位** | 深度分析：从逐字稿提取洞察 | 浅层关联：从洞察关联产品定义 |
| **分析深度** | 深（三层深度、证据编码、主题分析） | 浅（用户问题提取+证据强度判断） |
| **输入要求** | 高（需要逐字稿+调研目标） | 低（用研文档链接即可） |
| **输出复杂度** | 高（8节报告、证据账本、洞察卡） | 低（6节表格、以表格为主） |
| **覆盖用研包** | 第4部分✅ 第5部分✅ | 第4部分❌ 第5部分⚠️ |
| **产品关联** | ❌ 不关联 | ✅ 核心能力（Gap/需求/Idea） |
| **知识库依赖** | 无（纯分析） | 强（依赖ProductFact表） |
| **互补关系** | 提供洞察 → | 消费洞察，关联产品 |

### 关键发现

1. **两个技能是上下游关系**：用研逐字稿分析产出洞察 → 用研洞察拉通消费洞察并关联产品
2. **用研逐字稿分析** 是完整的逐字稿分析能力，覆盖用研包第4/5部分
3. **用研洞察拉通** 不做深度分析，核心价值在"产品Gap识别"和"需求/Idea映射"
4. **两个技能都不生成**：OnePage、用研需求文档、用研架构设计（用研包第1/2/3部分）

## 五、建议：技能模板调整方向

### 5.1 用研逐字稿分析与洞察报告

| 调整项 | 当前状态 | 建议 |
|--------|---------|------|
| 输出与用研包对齐 | 输出8节，与最终报告基本匹配 | ✅ 无需大改，当前输出已覆盖用研包第4/5部分的核心模块 |
| OnePage生成 | 不支持 | 如需支持，需新增独立技能或扩展现有能力 |
| 用研需求文档生成 | 不支持 | 建议新增：基于预检阶段提取的调研目标，自动生成用研需求文档模板 |
| 定量数据模块 | 作为交叉验证辅助 | 如需独立定量分析报告，需增强问卷/KANO数据的处理能力 |

### 5.2 用研洞察拉通

| 调整项 | 当前状态 | 建议 |
|--------|---------|------|
| 缺少证据账本 | 只有简单的"证据"列 | 建议增加：YAML格式证据条目（source/participant/type/content/confidence），与逐字稿分析技能的证据账本格式对齐 |
| 缺少主题分析 | 无 | 建议增加：至少支持频次×强度矩阵和分群分析，否则输出过于简单 |
| 缺少洞察深度 | 只有"用户问题"，无三层深度 | 建议增加：对每个核心问题要求至少达到中层深度（要什么+为什么） |
| 缺少反例检查 | 无 | 建议增加：对每个Gap标注是否存在反例或矛盾证据 |
| 缺少决策记录 | 无 | 建议增加：需要产品团队做决策的条目 |
| 输出过于表格化 | 6个表格 | 建议增加：核心发现的文字总结段落（不只是表格） |
| 与逐字稿分析的衔接 | 无显式衔接 | 建议增加：当输入是逐字稿分析的输出时，自动跳过重复的证据提取步骤 |

### 5.3 两个技能的协同优化

| 建议 | 说明 |
|------|------|
| **建立管道衔接** | 用研逐字稿分析的输出（洞察卡+决策记录）应可直接作为洞察拉通的输入 |
| **统一证据格式** | 两个技能的证据强度/置信度体系应统一（当前一个是HIGH/MEDIUM/LOW，一个是弱/中/强） |
| **合并为完整用研包** | 考虑将两个技能合并为一个"用研包生成"技能，一次性输出完整的5部分用研包 |
| **补充用研包第1-3部分** | 当前两个技能都不覆盖OnePage、用研需求文档、用研架构设计，如需完整用研包需新增能力 |

---

## 六、结论

| 维度 | 用研逐字稿分析与洞察报告 | 用研洞察拉通 |
|------|------------------------|-------------|
| 用研包第1部分（OnePage） | ❌ | ❌ |
| 用研包第2部分（用研需求文档） | ❌ | ❌ |
| 用研包第3部分（用研架构设计） | ❌ | ❌ |
| 用研包第4部分（逐字稿分析） | ✅ 高质量 | ❌ 不支持 |
| 用研包第5部分（最终报告） | ✅ 高质量 | ⚠️ 低质量（仅表格，缺深度） |
| 产品Gap关联 | ❌ | ✅ 核心能力 |
| 需求/Idea映射 | ❌ | ✅ 核心能力 |

**最终判断：两个技能互补，但都不完整覆盖用研包5部分。用研逐字稿分析覆盖第4/5部分质量高；用研洞察拉通的核心价值在产品关联而非用研分析。如需完整用研包生成能力，需要：(1) 将两个技能串联；(2) 补充第1-3部分的生成能力。**
