# 用研架构设计生成技能 (user-research-architecture-generator)

## 技能定位

从用研目标自动生成完整的用研架构设计文档，覆盖从研究目标定义到现场执行的全流程。
输入用研目标和产品信息，输出可直接执行的用研方案，包含访谈大纲/体验SOP、样本设计、问题框架、展示材料清单和执行SOP。

## 适用场景

- 新产品/功能的定性用户调研方案设计
- 竞品对比体验测试方案设计
- 产品内测/体验测试SOP设计
- 座谈会/深度访谈/体验测试的方案生成

## 输入要求

### 必填项

```yaml
research_objective: string        # 用研目标（一句话描述本次用研要验证什么）
product_name: string              # 产品名称
research_method: string           # research_method: 座谈会 | 体验测试 | 深度访谈 | 内测
```

### 选填项（有则提升输出质量）

```yaml
product_description: string       # 产品简介/卖点
target_users: string              # 目标用户画像
sample_size: number               # 样本量
duration_minutes: number          # 总时长（分钟）
competitors: list                 # 竞品列表
key_dimensions: list              # 需要测试的核心维度（如：外观/功能/成像/价格）
display_materials: list           # 已有的展示材料
validation_questions: list        # 需要验证的核心问题列表
existing_context: string          # 已有的调研数据、飞书文档链接等
```

## 处理流程

```
输入用研目标
→ Step 1: 解析研究类型（座谈会/体验测试/深度访谈/内测）
→ Step 2: 根据研究类型确定架构模板
→ Step 3: 生成核心验证问题（3-5个）
→ Step 4: 设计样本方案（人数、配比、筛选条件）
→ Step 5: 生成访谈大纲/体验SOP（按时间分板块）
  → 5a: 热身与画像（5-10%时长）
  → 5b: 核心体验/讨论（60-70%时长）
  → 5c: 产品测试/竞品对比（如有，20-30%时长）
  → 5d: 未来展望/总结（5-10%时长）
→ Step 6: 设计展示材料清单
→ Step 7: 生成执行SOP（招募→筛选→通知→现场→问卷→分析）
→ Step 8: 质量自检
→ Step 9: 输出完整用研架构设计文档
```

## 输出格式模板

### 输出结构总览

```markdown
# {产品名称} 用研架构设计

## 一、用研概览
- 用研目标
- 核心验证问题（3-5个）
- 研究方法
- 样本量与配比
- 总时长

## 二、用户招募与筛选
- 招募方式
- 筛选条件（人口统计 + 行为条件）
- 样本配比表
- 招募时间线

## 三、访谈大纲/体验SOP
### 板块一：热身与用户画像（Xmin）
- 目标
- 讨论问题
- 主持人注意事项

### 板块二：核心体验/讨论（Xmin）
- 目标
- 讨论问题（含追问路径）
- 主持人注意事项

### 板块三：产品测试/竞品对比（Xmin）
- 目标
- 测试维度
- 展示材料
- 讨论问题

### 板块四：未来展望与总结（Xmin）
- 目标
- 讨论问题

## 四、展示材料清单
- 材料名称
- 用途
- 准备要求

## 五、执行SOP
### 5.1 前期准备
- 人员配置
- 场地要求
- 设备清单

### 5.2 现场流程
- 到场签到
- 引导话术
- 体验流程
- 问卷填写

### 5.3 数据收集
- 问卷设计要点
- 记录要求

### 5.4 后续跟进
- 数据分析框架
- 输出物清单
- 时间节点

## 六、质量自检清单
```

### 板块详细格式

每个板块必须包含：

```markdown
### 板块N：{板块名称}（Xmin/Xmin累计）

**目标：** {本板块要达成的研究目标}

**讨论问题：**
1. {开放性问题}
   - 追问：{追问路径}
   - 主持人注意：{引导要点}

**展示材料：** {本板块需要的展示材料}

**主持人注意事项：**
- {注意事项1}
- {注意事项2}
```

### 样本配比表格式

```markdown
| 维度 | 条件 | 人数 | 占比 |
|------|------|------|------|
| 性别 | 男 | X | X% |
| 性别 | 女 | X | X% |
| 体型 | 正常&偏瘦 | X | X% |
| 体型 | 超重 | X | X% |
| 使用经验 | 有经验 | X | X% |
| 使用经验 | 无经验 | X | X% |
```

### 问题框架规范

每个问题必须包含：
- **问题主体**：开放性、非引导性
- **追问路径**：至少1条追问方向（标注a/b/c）
- **主持人注意**：引导要点、不引导禁区
- **关联维度**：该问题验证哪个核心问题

## 质量自检清单

输出完成后，按以下清单逐项检查：

### 结构完整性
- [ ] 是否包含用研概览（目标+方法+样本+时长）
- [ ] 是否包含用户招募与筛选方案
- [ ] 是否包含完整的访谈大纲/体验SOP
- [ ] 每个板块是否都有：目标、问题、追问、主持人注意
- [ ] 是否包含展示材料清单
- [ ] 是否包含执行SOP
- [ ] 板块时长是否合理分配，总计是否匹配

### 问题质量
- [ ] 问题是否开放、非引导
- [ ] 是否有追问路径
- [ ] 是否覆盖所有核心验证问题
- [ ] 问题顺序是否从浅入深
- [ ] 是否包含体验类问题（如适用）
- [ ] 是否包含对比测试问题（如适用）

### 样本设计
- [ ] 样本量是否合理
- [ ] 配比是否覆盖关键维度
- [ ] 筛选条件是否可执行
- [ ] 是否有参考依据

### 执行可行性
- [ ] 展示材料是否可准备
- [ ] 时间分配是否合理
- [ ] 现场流程是否清晰
- [ ] 人员配置是否明确
- [ ] 问卷/数据收集是否设计

### 表达规范
- [ ] 语言是否口语化（适合主持人念）
- [ ] 是否避免专业术语
- [ ] 追问标注是否统一（a/b/c）
- [ ] 板块时长格式是否统一（Xmin/累计min）

## 质量评分标准

| 维度 | 权重 | 100分标准 | 90分标准 | 80分标准 |
|------|------|-----------|----------|----------|
| 结构完整性 | 25% | 覆盖全部7个模块，无遗漏 | 覆盖6个模块，个别细节缺失 | 覆盖5个模块 |
| 问题设计 | 30% | 问题开放、追问完整、覆盖所有验证点 | 问题基本开放、追问较完整 | 部分问题引导性较强 |
| 样本设计 | 15% | 配比科学、有依据、可执行 | 配比合理、基本可执行 | 配比基本合理 |
| 展示材料 | 10% | 材料清单完整、用途明确 | 材料清单较完整 | 材料清单有遗漏 |
| 执行SOP | 10% | 流程清晰、人员明确、可直接执行 | 流程基本清晰 | 流程有模糊处 |
| 表达规范 | 10% | 口语化、格式统一、标注规范 | 基本口语化 | 部分书面化 |

## 示例输入

```yaml
research_objective: 了解用户对拍立得的购买决策、使用体验及对新产品的外观/功能/成像偏好
product_name: 拍立得
research_method: 座谈会
product_description: 小米拟进入拍立得市场，需了解用户需求和产品方向
target_users: 拍立得现有用户，18-40岁
sample_size: 8
duration_minutes: 120
competitors: 富士、宝丽来、徕卡
key_dimensions:
  - 外观偏好
  - 功能需求
  - 成像效果
  - 价格接受度
validation_questions:
  - 用户购买拍立得的核心动机和决策因素是什么
  - 用户对现有产品的满意点和痛点是什么
  - 用户对外观/功能/成像的偏好排序
  - 用户对小米进入该市场的态度
```

## 注意事项

1. **问题必须开放**：避免"你是否喜欢XX"式封闭问题，改用"你觉得XX怎么样"
2. **追问必须具体**：不能只写"追问细节"，要写明追问方向
3. **主持人注意必须区分**：哪些要主动引导追问，哪些绝不能引导
4. **展示材料必须关联**：每份材料对应哪个问题，用途是什么
5. **时间必须合理**：热身不超过10%，核心体验占60-70%
6. **样本必须可执行**：筛选条件不能太理想化，要考虑实际招募难度
7. **SOP必须落地**：包含具体话术、时间节点、人员分工
8. **功能细分必须到位**：涉及功能测试时，要列出具体功能清单（分层级，如中端/高端），每个功能要有独立的追问路径
9. **场景细分必须具体**：涉及使用场景时，要列出具体场景列表（如日常记录、旅游、聚会等），不要笼统说"各种场景"
10. **对比维度必须明确**：涉及竞品对比时，要明确对比哪些维度（外观/功能/成像/价格），每个维度要有具体问题
